Achados Críticos (P0-P1)

P0-01: Integração CRM é uma mentira

O spec diz "conectado 100% ao CRM — mesmas tabelas, mesmas funcionalidades". A realidade:

  • fn_restricoes_listar aceita p_lead_id — funciona. Mas o frontend MdZap usa LeadRestricoesMdzapTab que chama CrmAuditoriaPanel sem o callback onRestricoesCadastradas. Ao cadastrar restrições, as queries não invalidam.
  • processos_documentos tem lead_id, restricao_id, origem (inclui 'whatsapp_chat'). Perfeito. Mas o MdZap nunca insere docs aqui — os docs ficam no Supabase Storage sem vínculo à tabela.
  • distribuir_lead_completo exige coluna_crm = 'triagem'. Mas o MdZap cria leads em 'em_atendimento'. O lead precisaria ser movido manualmente para 'triagem' antes de distribuir. Ninguém conectou esses fluxos.
  • A view view_crm_leads_kanban filtra arquivado = false — leads do MdZap que são arquivados no MdZap (card 3 pontinhos → arquivar) não somem do CRM porque o campo arquivado pode não ser atualizado pelo webhook.

P0-02: pending_lead_data — Nunca Usado

O spec v2 define que o Doc Reader popula whatsapp_conversas.metadata.pending_lead_data antes do cadastro formal. Resultado em produção:

SELECT count(*) FROM whatsapp_conversas
WHERE metadata->'pending_lead_data' IS NOT NULL → 0 registros

O campo JSONB existe, o tipo TypeScript PendingLeadData existe, mas nenhum webhook n8n seta esse campo. O formulário pré-preenchido que o frontend mostra está vazio.

P0-03: leads.awaiting_human — Dois campos, zero confiança

A tabela leads tem DOIS campos para a mesma coisa:

CampoTipoNullableDefault
awaiting_humanbooleanYESnull
aguardando_humanobooleanNOfalse

O trigger trg_sync_awaiting_human sincroniza os dois. Problema: awaiting_human é nullable — pode ser null, true ou false. O COALESCE em fn_mdzap_conversas trata isso, mas qualquer query direta na tabela leads que use WHERE awaiting_human = true vai ignorar nulls. E a Edge Function seta metadata.awaiting_human na conversa, não no lead — são 3 lugares diferentes para a mesma informação.

P1-01: Documentos do WhatsApp não vão para processos_documentos

A Edge Function mdzap-webhook baixa mídia e salva no bucket mdzap-media. A URL fica em whatsapp_mensagens.media_url. Mas:

  • processos_documentos é a tabela oficial de documentos do CRM
  • Tem lead_id, restricao_id, categoria, tipo_documento, origem (com CHECK 'whatsapp_chat')
  • Nenhum processo copia os docs do bucket para esta tabela
  • O LeadDocumentosTab do MdZap provavelmente consulta uma tabela diferente do que o ClienteDrawerTabDocumentos do CRM

Resultado: docs do WhatsApp vivem num bucket isolado. CRM não vê. MdZap não mostra docs do CRM.

P1-02: fn_mdzap_conversas — 6 reescritas em 20 dias

Essa RPC foi reescrita 6 vezes entre 2026-03-19 e 2026-04-07:

  1. v1: SQL simples, TABLE return
  2. v2: Adicionou aguardando_humano
  3. v3: plpgsql + soft delete
  4. v4: Paginação + search + JSONB envelope
  5. v5: Drop overload para fix PGRST203
  6. v6: Fix awaiting_human COALESCE + add lead_cpf + agente_slug

6 reescritas geram risco de regressão. A assinatura mudou de (uuid, uuid) para (uuid, int, int, uuid, text, text). Qualquer cache PostgREST pode estar desatualizado.

P1-03: mdzap_message_queue — 20 registros órfãos

A tabela mdzap_message_queue tem 20 registros com expires_at DEFAULT now()+5min. Não existe nenhum processo que limpa registros expirados. Não existe nenhum consumer que processa a fila. A tabela existe mas não tem propósito funcional visível.

Tabela leads — Schema Real (64 colunas)

A tabela leads é a mais importante do sistema. Ela é compartilhada entre CRM e MdZap. Tem 64 colunas — das quais pelo menos 15 são legado/redundantes.

Campos Usados pelo MdZap

  • nome, cpf, rg, whatsapp
  • data_nascimento, sexo, profissao
  • endereco_* (logradouro, numero, complemento, bairro, cidade, uf, cep, estado)
  • coluna_crm, awaiting_human
  • advogado_id, atendente_id
  • lead_id → whatsapp_conversas.lead_id

Campos Redundantes / Legado MdZap

  • mdzap_contact_id — TEXT, indexado. Duplica whatsapp_conversas.contato_numero
  • mdzap_conversation_id — TEXT, indexado. Duplica whatsapp_conversas.id
  • wa_conversation_id — TEXT, indexado. Terceiro campo para conversation ID
  • wa_status, wa_blocked, wa_unread_count, wa_last_message_at — duplicam dados de whatsapp_conversas
  • last_inbound_at, last_outbound_at — duplicam whatsapp_conversas.ultimo_msg_at
  • aguardando_humano — duplica awaiting_human (sync via trigger)
  • restricoes (JSONB) — duplica tabela restricoes
  • documentos (JSONB) — duplica tabela processos_documentos

São 10+ colunas redundantes — schema drift entre "dados no lead" vs "dados nas tabelas relacionais".

O Que Está Bem Feito

  • Edge Function mdzap-webhook (525 linhas) — bem escrita. Normaliza Meta + Z-API, retry com exponential backoff, download mídia, fallback para media_id_failed. Código sólido.
  • RLS em 100% das tabelas — todas as 20 tabelas têm tenant isolation. SECURITY DEFINER em todas RPCs.
  • Soft delete — aplicado nas tabelas que importam (conversas, mensagens, lead_memory, templates, notas).
  • updated_at triggers — 15 de 20 tabelas têm trigger automático.
  • CHECK constraints — TEXT CHECK (nunca ENUM) em todas as tabelas que precisam. Padrão MdFlow respeitado.
  • fn_link_conversa_to_lead — bem implementada com validação de org cruzada. Pronta para uso.
  • Indexes compostos — idx_wa_msg_org_conversa_created (organizado para a query mais comum). GIN no metadata.
  • whatsapp_followups — schema completo com tentativas, max_tentativas, scheduled_at, executed_at. Bem desenhado, só falta o consumer.
  • ICP análises — 3 registros, webhook funciona, cron ativo. Único subsistema IA funcionando de verdade.

Recomendações (Prioridade)

1. Fechar o gap CRM (P0 — antes de qualquer outra coisa)

O spec promete integração total com CRM. Na prática, os dados vivem em mundos separados. Prioridade absoluta:

  1. Frontend: chamar fn_link_conversa_to_lead no momento do cadastro
  2. Frontend: passar onRestricoesCadastradas no CrmAuditoriaPanel
  3. n8n: criar webhook que copia docs aprovados para processos_documentos
  4. n8n: popular pending_lead_data quando Doc Reader extrai dados

2. Ativar os subsistemas mortos (P1)

Três tabelas bem desenhadas com zero registros:

  • mdzap_lead_memory — precisa do webhook de memória ativo
  • whatsapp_followups — precisa do consumer cron
  • pending_lead_data — precisa do Doc Reader populando

3. Limpar a tabela leads (P2)

64 colunas. 10+ redundantes com tabelas relacionais. Cada nova feature adiciona mais. Recomendo:

  • Auditar uso de mdzap_contact_id, wa_*, restricoes JSONB, documentos JSONB
  • Migrar qualquer leitura para as tabelas relacionais corretas
  • Marcar deprecated e remover em iteração futura

Conclusão

5.2/10 — "Fundação sólida, casa vazia."

O schema está bem desenhado. RLS, soft delete, triggers, indexes compostos, CHECK constraints — tudo correto. A Edge Function é código de qualidade. O problema não é a qualidade do que existe, é a quantidade do que não existe.

Os 3 subsistemas que fariam o MdZap funcionar de verdade (pending_lead_data, lead_memory, followups) têm zero registros. A integração com CRM, que é o ponto central do módulo, é superficial — os componentes foram copiados sem conectar as queries. E a tabela leads com 64 colunas, 10+ redundantes, é uma bomba-relógio de schema drift.

A nota subiria para 7-8/10 com os 4 fixes P0 implementados e os subsistemas ativados. A fundação está lá. Falta construir em cima dela.
