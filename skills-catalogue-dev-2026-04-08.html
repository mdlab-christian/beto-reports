/page-loop — Orquestrador de Qualidade v3.0
O maestro de todas as skills
/page-loop {PAGINA}
ORQUESTRADOR v3.0 3-4h

O que faz

Orquestrador completo de 5 fases. Garante que cada fase só inicia se a anterior passou. P0 bloqueante encontrado → resolve antes de avançar. Máximo 2 loops de QA para evitar diminishing returns. Único HTML aberto no browser é o loop-report final.

Sintaxe

/page-loop clientes # todas as fases /page-loop clientes --from fase2 # começa na fase 2 /page-loop clientes --only fase1,fase3 # só fases listadas + finalize /page-loop clientes --skip fase3 # pula fase 3

5 Fases

0
Pre-flight — valida ambiente antes de gastar tokens
7 checks: spec, dev server, TypeScript, Supabase, git, reports, carregamento
1
Backend First — schema-first para evitar falsos positivos no QA
backend-tester + backend-executor
2
QA Funcional — loop máx 2x, 17 grupos de testes
qa-tester + executor, parar após P0 = 0
3
Deep Front — UI/UX profunda, 9 categorias, 5 breakpoints
deep-front-audit + deep-front-executor + front-tester
4
Palavra Final — senior review com contexto máximo, veredicto PRONTA/QUASE/NÃO PRONTA
senior-review + senior-executor

Veredicto final

PRONTA
nota ≥ 8.5, zero P0, ≤2 P1
QUASE
nota 7.0-8.4, zero P0, P1 docs
NÃO PRONTA
nota < 7.0 ou P0 não resolvidos

Skills de Execução
Corrigem o que as auditorias encontraram. 1 commit por bug.
/page-executor
EXECUÇÃOv4.0
Lê o QA report → corrige P0→P1→P2 → 1 commit por bug → verifica → log HTML → push. Inclui: silent-drop check, cross-page impact, TypeScript verify após cada fix, spec-aware.
  • Apresenta lista de bugs antes de executar (confirmação visual)
  • Ordem: P0 → P1 → P2 → dependências antes de dependentes
  • Migrations SQL antes dos componentes que dependem delas
  • Verifica cross-page impact antes de alterar componente compartilhado
/page-backend-executor
EXECUÇÃO
Aplica migrations SQL, cria RPCs, corrige RLS via Supabase MCP. Loop até nota ≥ 9 (máx 3 ciclos). Usa Management API para DDL.
  • Usa mcp__claude_ai_Supabase__apply_migration para DDL
  • RETURNING id em todo INSERT
  • TEXT CHECK em vez de ENUM nativo
  • SECURITY DEFINER em RPCs tenant-aware
/page-deep-front-executor
EXECUÇÃO
Corrige achados do deep-front-audit. Ordem: CSS → JSX → Funcional. Loop interno até nota ≥ 9 (máx 3 ciclos). Screenshot como evidência após cada fix visual.
  • Nunca hardcode de cor — sempre tokens CSS var(--token)
  • hex → CSS vars obrigatório
  • Glassmorphism apenas em cards/modais, nunca tabelas/inputs
/page-senior-executor
EXECUÇÃOSÊNIOR
Executa TODOS os achados do senior-review — backend + frontend + n8n no mesmo ciclo. Commit por fix. Mais poderoso e abrangente que executor padrão.
  • Corrige banco, código e workflows n8n no mesmo ciclo
  • Usa contexto máximo do senior-review
  • Prioridade: backend → frontend → n8n → documentação

O que falta no pipeline atual
Gaps identificados que nenhuma skill atual cobre bem
Esta análise é baseada no que o CC observa rodando as skills dia a dia no MdFlow. São os pontos onde erros "escapam" mesmo após rodar o page-loop completo.
❌ GAP-01 — Lógica de negócio jurídica não é testada
Nenhuma skill valida se a lógica do domínio jurídico está correta. Ex: um processo pode ser distribuído sem ter restrição vinculada? Um acordo pode ser criado sem processo ativo? O campo "valor da causa" pode ficar vazio em distribuição? A skill atual testa "o botão funciona" mas não "faz sentido para o negócio".
Exemplo: /acordos aceita criar acordo sem processo_id → dado inválido no banco mas sem erro de UI
❌ GAP-02 — Fluxo Lead-to-Cash não é testado end-to-end
O pipeline /crm → /clientes → /distribuicao → /processos → /controller → /decisoes → /acordos → /financeiro → /execucoes nunca é testado como fluxo contínuo. Cada página é testada isoladamente. Quebras no handoff entre páginas (ex: lead convertido no CRM não aparece em /clientes) nunca são detectadas pelas skills atuais.
Exemplo: processo criado em /distribuicao não aparece no /controller porque falta campo distribuicao_status
❌ GAP-03 — Cegueira financeira não é sistematicamente testada
O padrão acesso_financeiro=false → R$••••• deve ser respeitado em 6+ páginas (/financeiro, /acordos, /execucoes, /pagina_processo, /audiencias, /dashboard). Nenhuma skill atual troca o perfil para um usuário sem acesso_financeiro e verifica sistematicamente que os valores foram mascarados.
Exemplo: /pagina_processo exibe valor da causa mesmo com acesso_financeiro=false
❌ GAP-04 — Testes de permissão por perfil
Perfis do sistema: juridico, comercial, admin, assinante, partner. Nenhuma skill testa sistematicamente que perfil X não vê o que é de Y. Um perfil comercial não deveria ver /controller. Um perfil jurídico não deveria ver /crm. Esses acessos são validados?
Exemplo: usuário comercial consegue acessar /processos/uuid diretamente pela URL mesmo sem permissão
⚠️ GAP-05 — n8n: cenários de erro não são simulados
As skills testam o happy path dos webhooks n8n. Mas e quando o n8n retorna success:false? E quando o webhook dá timeout? E quando o n8n retorna action="toast_error"? A UI trata todos esses cenários? Nenhuma skill simula falha intencional para validar o tratamento de erro.
⚠️ GAP-06 — Dados reais de teste nunca são limpos
Cada ciclo de QA cria registros com prefixo [QA] no banco. Após meses de ciclos, o banco de teste tem dezenas de registros espúrios que poluem os testes seguintes. Nenhuma skill tem uma fase de cleanup sistemático pós-ciclo.
⚠️ GAP-07 — Cálculos monetários não são validados matematicamente
O banco guarda valores em centavos INT8. O frontend converte para float para exibir. Nenhuma skill verifica que soma de parcelas = total do acordo, que o cálculo de honorários está correto, ou que valores não sofrem float imprecision no arredondamento.
🔵 GAP-08 — Componentes compartilhados entre páginas nunca são testados como unidade
ClienteDrawer é usado em /crm, /clientes e /distribuicao com comportamentos diferentes por contexto. ProcessoDrawer em /processos e /pagina_processo. RestricaoPastaCard em /clientes e /distribuicao. Uma mudança em um contexto pode quebrar outro — nunca testado em sequência.
🔵 GAP-09 — Spec vs implementação drift não é rastreado ao longo do tempo
A spec existe nos docs/ mas evolui. A implementação também. Nenhum mecanismo compara versão atual da spec vs estado atual do código e gera um diff histórico. O backend-tester faz isso uma vez, mas não de forma contínua.

Proposta: /page-complete-review
A skill que fecha todos os gaps — sem deixar passar nada
NOVA SKILL
page-complete-review

Uma skill que vai além do ciclo técnico atual. Enquanto o page-loop verifica "o código está correto?", esta skill pergunta "a página faz sentido para o negócio?", "todos os perfis veem o que deveriam?", "o fluxo completo funciona de ponta a ponta?". É a revisão que um dev sênior com conhecimento jurídico faria.

Diferencial principal
  • Valida lógica de negócio jurídica, não só código
  • Testa perfis de permissão sistematicamente
  • Cegueira financeira como test case obrigatório
  • Simula falhas de n8n (success:false, timeout)
  • Valida cálculos monetários matematicamente
  • Testa fluxo upstream/downstream da página
O que NÃO é
  • Não substitui page-loop — é complementar
  • Não re-audita UI/UX (deep-front-audit já faz)
  • Não re-testa schema (backend-tester já faz)
  • Não é mais lenta que o ciclo atual — é mais inteligente
  • Não executa correções — só audita (tem executor separado)
Quando usar
  • Após page-loop PRONTA — validação de negócio
  • Antes de liberar feature para produção
  • Após mudança de regra de negócio importante
  • Quando Christian reporta "algo está errado" sem saber o quê
  • Revisão trimestral de páginas críticas (/processos, /acordos)
Inputs necessários
  • Nome da página
  • Spec da página (já tem)
  • Glossário jurídico (~/.claude/mdflow-glossario.md)
  • Mapa de perfis e permissões (CLAUDE.md)
  • Regras de negócio da página (a descrever)

Spec Detalhada — /page-complete-review
O que deve ser implementado em cada bloco da skill
BLOCO 1 — Business Logic Validator
◆
Regras de pré-condição — "quais dados precisam existir antes desta ação ser possível?"
Ex: /distribuicao → processo precisa ter cliente + restrição + documentos Tier1. Se não tem, deve bloquear ou avisar?
◆
Regras de status state machine — "quais transições de status são válidas?"
Ex: processo em status "aguardando_documentos" pode ir para "em_analise"? Pode pular para "distribuido"?
◆
Regras de integridade entre registros — "um acordo sem processo vinculado é válido?"
Verificar FK constraints na prática, não só no schema
◆
Regras de prazos jurídicos — "campos de prazo são calculados corretamente?"
data_vencimento = data_distribuicao + prazo_legal. Está correto? Conta dias úteis?
BLOCO 2 — Permission Matrix Tester
◆
juridico — pode ver: processos, controller, decisoes, acordos (mascarado $), distribuicao. Não pode: /crm completo, /financeiro
◆
comercial — pode ver: /crm, /clientes, /indicacoes. Não pode: /controller, /decisoes, /processos (sensitive)
◆
admin — pode ver tudo. Verificar que config de tenant é inacessível para outros perfis.
◆
acesso_financeiro=false — 6 páginas devem mascarar R$: verificar cada uma com screenshots evidência.
◆
acesso_gestor=true/false — KPIs de equipe, relatórios gerenciais — visíveis apenas com flag ativa.
BLOCO 3 — Flow Integration Test
◆
Upstream check — "de onde vêm os dados desta página?"
Ex: /processos recebe dados de /distribuicao. Verificar que distribuicao_lote_id está preenchido corretamente.
◆
Downstream check — "o que esta página alimenta?"
Ex: /clientes alimenta /distribuicao. Verificar que campos necessários para a distribuição estão presentes.
◆
Handoff test — criar registro nesta página → navegar para a próxima → dados chegaram?
Lead convertido no CRM → aparece em /clientes com status correto?
BLOCO 4 — Financial Integrity
◆
Centavos vs float — banco guarda INT8 centavos, UI exibe float. Verificar arredondamento nos 4 casos: R$0,01 / R$999,99 / R$1.000,00 / R$99.999,99
◆
Soma de parcelas — soma de all parcelas = valor total do acordo? Verificar via query.
◆
Cegueira financeira completa — R$••••• em TODOS os contextos para usuário sem acesso_financeiro: listagem, drawer, PDF, export, toast, email
BLOCO 5 — n8n Failure Scenarios
◆
success:false — forçar cenário de erro no n8n. Frontend exibe toast_error? Não navega?
◆
Timeout — webhook demora 30s. Frontend tem loading state? Não duplica a requisição?
◆
action inválido — n8n retorna action desconhecido. Frontend não quebra (switch default)?
◆
Duplo clique — clicar 2x no botão de webhook. Duas requisições? Proteção contra?
BLOCO 6 — EPROC / Multi-tribunal (páginas jurídicas)
◆
CNJ format — número CNJ exibido em JetBrains Mono, copiável com click, formato NNNNNNN-DD.AAAA.J.TR.OOOO
◆
Tribunal detectado — TJRS / TJSC / TJSP exibidos corretamente baseado no CNJ
◆
OAB responsável — processo TJRS aparece para OAB RS (Christian ou Heloísa), não mistura com SC/SP
BLOCO 7 — Data Cleanup (pós-ciclo)
◆
Listar todos os registros criados durante o ciclo com prefixo [QA] ou [TESTE]
◆
Soft-delete ou hard-delete (confirmar com usuário) dos registros de teste
◆
Verificar que o cleanup não deixou orphans (FK violation)

Esta seção é o briefing mais valioso para o Claude Web planejar a nova skill. São insights que só aparecem com uso real do sistema.
🎯 Contexto jurídico não está nas skills
O CC sabe que é um sistema jurídico, mas as skills não têm embutido o conhecimento de quando uma restrição é "válida" para distribuição, quais são os documentos obrigatórios por tipo de ação, ou o que é uma intimação com prazo penal. O glossário existe (~/.claude/mdflow-glossario.md) mas não é carregado nas auditorias.
🔗 Cross-page testing é zero
Cada skill abre uma página, testa, fecha. Nenhuma skill abre /crm → cria lead → converte → vai para /clientes → adiciona restrição → vai para /distribuicao → inicia distribuição → verifica em /processos. Esse fluxo quebra silenciosamente e ninguém percebe.
👥 Multi-user testing inexistente
Sempre testamos com um único usuário. Nunca testamos: dois usuários editando o mesmo processo simultâneo. Realtime subscription cancela corretamente quando o user2 salva e o user1 ainda tem o drawer aberto? O dirty-state alerta antes ou depois do realtime update?
📊 Performance nunca medida
Nenhuma skill verifica quanto tempo leva para /processos carregar 5.800 registros. Se a query demora 3 segundos, o usuário vê loading eterno. staleTime está configurado mas nunca verificamos se o cache está realmente sendo usado.
🏷️ Estados de borda não têm spec
O que acontece quando um processo tem 0 restrições? Quando um cliente tem 47 restrições? Quando um acordo tem 120 parcelas? As specs documentam o happy path mas não os extremos. As skills testam com dados "normais" — os extremos quebram silenciosamente.
🔄 Ciclos de lapidação perdem contexto
Cada novo ciclo de page-loop "esquece" o que foi decidido no ciclo anterior. Por exemplo: "não corrija X pois é comportamento intencional do negócio" precisa ser redito toda vez. O arquivo docs/revisoes/{PAGINA}.md existe mas não é um "livro de decisões" — é só uma lista de pendências.
Sugestão de nome e posicionamento: A nova skill deve se chamar /page-complete-review ou /page-logic-review. Ela não substitui o page-loop — ela complementa. Roda APÓS o page-loop estar PRONTA, como uma "auditoria de negócio" antes do deploy em produção. É o que diferencia "código correto" de "produto correto".

Perguntas para o Claude Web ao planejar
Pontos de decisão que precisam ser definidos no briefing
DECISÃO 1 — Escopo
A skill cobre UMA página ou o FLUXO completo? Se for uma página: precisa saber seu contexto upstream/downstream. Se for o fluxo: precisa mapear quais páginas estão no caminho de cada fluxo.
DECISÃO 2 — Fonte das regras de negócio
As regras jurídicas ficam hardcoded na skill? No glossário? Em arquivo separado por página? Sugestão: criar docs/regras-negocio/{PAGINA}.md como SSOT e a skill carrega esse arquivo.
DECISÃO 3 — Perfis de teste
A skill precisa de usuários de teste pré-criados por perfil no banco? Sugestão: criar 5 usuários fixos (um por perfil) com email padrão qa.juridico@, qa.comercial@, etc.
DECISÃO 4 — Integração com page-loop
Deve ser uma fase extra do page-loop (fase 5)? Ou skill independente? Sugestão: skill independente que pode ser acionada via /page-loop --with-complete ou standalone.
DECISÃO 5 — Modelo de IA
As regras de negócio jurídicas são complexas. Usar Opus 4.6 para a análise de lógica? Ou Sonnet 4.6 com contexto bem estruturado é suficiente?
DECISÃO 6 — Executor separado?
Criar também um /page-complete-executor que corrige os achados? Ou os achados são redirecionados para page-executor / page-backend-executor existentes?
