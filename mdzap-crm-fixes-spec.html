Backend (B01-B07) — Executar Primeiro

B01 fn_mdzap_conversas: filtrar lead.deleted_at P1 BUG

Problema

LEFT JOIN com leads nao filtra deleted_at. Leads deletados aparecem nas conversas MdZap.

Localizacao

RPC fn_mdzap_conversas — versao mais recente em migrations

Fix SQL

-- ANTES: LEFT JOIN leads l ON l.id = wc.lead_id -- DEPOIS: LEFT JOIN leads l ON l.id = wc.lead_id AND l.deleted_at IS NULL

Cross-Impact

  • Conversas SEM lead (lead_id IS NULL) continuam aparecendo — OK
  • view_crm_leads_kanban JA filtra deleted_at — sem divergencia apos fix
  • useMdzapConversas() para de mostrar leads fantasma — melhora UX
  • MdzapBadgeStatus em CrmLeadCard: se lead deletado, conversa some do MdZap mas lead ja nao existe no CRM — consistente

Risco

ZERO — apenas remove dados invalidos da query

B02 whatsapp_mensagens: habilitar RLS P1 SEGURANCA

Problema

Policies estao comentadas em 20260407000002_mdzap_rls_mensagens.sql. Tabela sem RLS em producao.

Fix SQL

ALTER TABLE whatsapp_mensagens ENABLE ROW LEVEL SECURITY; -- Policy via conversa (mensagem pertence a conversa que pertence a org) CREATE POLICY "msg_select" ON whatsapp_mensagens FOR SELECT USING ( EXISTS ( SELECT 1 FROM whatsapp_conversas wc WHERE wc.id = whatsapp_mensagens.conversa_id AND wc.organizacao_id = public.get_org_id() ) ); -- Repetir para INSERT/UPDATE

Cross-Impact

  • useMdzapMessages hook: ja filtra por conversa_id — sem quebra
  • MdzapMsgThread: renderiza mensagens da conversa atual — sem quebra
  • n8n webhooks: usam service_role — bypassam RLS — sem quebra
  • Performance: JOIN com whatsapp_conversas em toda query — criar indice em conversa_id

Risco

BAIXO — testar com usuario normal (nao service_role) antes de deploy

B04 Triggers set_updated_at faltantes P2 CONSISTENCIA

Problema

Tabelas com coluna updated_at mas sem trigger automatico.

Tabelas

  • whatsapp_conversas — tem updated_at column, sem trigger
  • whatsapp_numeros — tem updated_at column, sem trigger
  • leads_origens — mencionada em JOINs, sem trigger

Fix SQL

CREATE TRIGGER set_updated_at BEFORE UPDATE ON whatsapp_conversas FOR EACH ROW EXECUTE FUNCTION update_updated_at_column(); CREATE TRIGGER set_updated_at BEFORE UPDATE ON whatsapp_numeros FOR EACH ROW EXECUTE FUNCTION update_updated_at_column(); CREATE TRIGGER set_updated_at BEFORE UPDATE ON leads_origens FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

Cross-Impact

  • Realtime subscriptions em whatsapp_conversas: ouvem postgres_changes — trigger nao afeta
  • RPCs que fazem UPDATE: trigger aplica APOS update — sem quebra
  • CRM: leads_origens trigger so muda updated_at — CrmOrigensChart nao depende desse campo

Risco

ZERO — trigger standard do MdFlow

B05 CrmOrigensChart: RPC server-side P1 PERFORMANCE

Problema

CrmOrigensChart.tsx faz SELECT * FROM leads + client-side Map aggregation. Com 5.800+ leads, transfere milhares de rows.

Fix SQL

CREATE OR REPLACE FUNCTION fn_crm_leads_por_origem(p_org_id UUID) RETURNS TABLE (origem_nome TEXT, total BIGINT) LANGUAGE sql SECURITY DEFINER STABLE AS $$ SELECT COALESCE(lo.nome, 'Sem origem'), COUNT(*) FROM leads l LEFT JOIN leads_origens lo ON l.origem_id = lo.id WHERE l.organizacao_id = p_org_id AND l.deleted_at IS NULL AND l.arquivado = false GROUP BY lo.nome ORDER BY COUNT(*) DESC; $$;

Frontend

Trocar query direta por supabase.rpc('fn_crm_leads_por_origem', { p_org_id: orgId })

Cross-Impact

  • CrmOrigensChart e o UNICO consumer — sem quebra em outros
  • MdZap: nao usa esse chart — sem impacto
  • Dashboard: se tiver widget de origens, apontar para mesma RPC

Risco

BAIXO — testar shape de retorno no frontend

B06 CrmTrendChart: RPC server-side P1 PERFORMANCE

Problema

CrmTrendChart.tsx faz SELECT created_at FROM leads WHERE >= 90d + client-side Map por semana.

Fix SQL

CREATE OR REPLACE FUNCTION fn_crm_trend_chart(p_org_id UUID, p_days INT DEFAULT 90) RETURNS TABLE (semana TEXT, total BIGINT) LANGUAGE sql SECURITY DEFINER STABLE AS $$ SELECT to_char(date_trunc('week', l.created_at), 'YYYY-MM-DD'), COUNT(*) FROM leads l WHERE l.organizacao_id = p_org_id AND l.deleted_at IS NULL AND l.created_at >= now() - make_interval(days => p_days) GROUP BY date_trunc('week', l.created_at) ORDER BY date_trunc('week', l.created_at); $$;

Cross-Impact

  • CrmTrendChart e o UNICO consumer — sem quebra
  • MdZap: nao usa esse chart — sem impacto

Risco

BAIXO
