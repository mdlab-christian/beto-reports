Decisoes Pendentes — CTO Christian

Bloqueantes para iniciar implementacao
Todas as decisoes abaixo precisam de resposta do Christian antes de rodar DDL ou criar nos n8n. Nenhuma e tecnicamente incorreta — e uma escolha de design.
PENDENTE CTO DEC-01
Como diferenciar os SDRs de acordos — por agente_slug ou por numero_id?
Opcao A — agente_slugs separados (RECOMENDADA)
Criar sdr_acordos_cp e sdr_acordos_hm em mdzap_ai_modules. Cada numero tem seu proprio slug.
+ Mais granular, permite prompts diferentes por advogado. + Consistente com arquitetura atual. + Nao precisa alterar logica de resolucao de agente.
Opcao B — manter sdr_acordos unico + filtrar por numero_id
Um unico slug sdr_acordos, o frontend diferencia pela conversa.numero_id.
+ Mais simples no banco. - Complica o frontend e o n8n (precisa de logica adicional). - Prompt unico para dois advogados diferentes.
PENDENTE CTO DEC-02
Os dois novos numeros ja estao registrados na Meta Business API?
Impacto
Sem o phone_number_id real da Meta, nao e possivel registrar em whatsapp_numeros nem configurar o n8n. Se nao estiverem registrados, Christian precisa criar os numeros no Meta Business Manager antes da implementacao tecnica.
PENDENTE CTO DEC-03
Como vincular numero → advogado no banco?
Opcao A — campo advogado_oab em whatsapp_numeros
Adicionar coluna advogado_oab TEXT na tabela whatsapp_numeros.
+ Simples, consultavel. + JOIN direto com tabela de processos para filtrar por advogado.
Opcao B — metadata JSONB
Usar campo metadata JSONB existente: {"advogado_oab": "RS116571"}
+ Zero DDL. - Menos performatico para queries. - Nao tipado.
Opcao C — vincular via agente_slug (RECOMENDADA se DEC-01=A)
O proprio agente_slug (sdr_acordos_cp / sdr_acordos_hm) ja identifica o advogado. Nao precisa de campo extra.
PENDENTE CTO DEC-04
Painel lateral de acordos — quais campos exatamente?
Campos levantados pelo Christian
Empresa credora | Escritorio (vara) | CNJ do processo | Status do processo | Valor do acordo em negociacao | Outros campos necessarios?
Origem dos dados
De onde vem a vinculacao conversa ↔ processo? O lead da conversa ja e o cliente no MdFlow? Ou precisa de vinculo manual?
PENDENTE CTO DEC-05
Frontend: ampliar SdrModeToggle ou criar novo filtro por advogado?
Opcao A — Ampliar SdrModeToggle com sub-filtro
[ Leads ] [ Triagem ] [ Acordos ▾ ] → dropdown: Todos / Christian / Heloisa
+ Nao muda o layout existente. + Consistente com UX atual.
Opcao B — Tab separado por advogado
[ Leads ] [ Triagem ] [ Acordos CP ] [ Acordos HM ]
+ Mais explicito. - Pode ficar apertado no mobile com 4+ tabs.

Dependencias Criticas e Riscos

Risco 1 — Edge Function nao extrai phone_number_id
Se a Edge Function mdzap-webhook nao armazena o phone_number_id ao inserir em whatsapp_mensagens, toda a arquitetura multi-numero falha. Precisa ser auditada ANTES de qualquer outro desenvolvimento.
Risco 2 — fn_mdzap_conversas nao filtra por array de slugs
Se a RPC so aceita um slug por vez, a visao unificada "Acordos" (mostrando CP + HM juntos) nao funciona. Precisaria de adaptacao na funcao ou chamadas paralelas no frontend.
Ponto positivo — Arquitetura multi-numero ja existe
whatsapp_numeros suporta N numeros. fn_mdzap_conversas aceita p_numero_id. A base esta pronta; o trabalho e configurar e conectar os novos numeros.
Ponto positivo — SdrModeToggle ja filtra por slug
O componente de toggle ja funciona com agente_slug. Ampliar para aceitar array de slugs ou sub-filtro e uma mudanca incremental, nao uma reescrita.
