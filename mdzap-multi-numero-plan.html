Regra de ouro — definida pelo CTO
Cada agente SDR e um "mundo" separado: dashboard proprio, chat proprio, painel lateral proprio, permissoes proprias, numero WhatsApp proprio, prompt proprio. Nenhum agente se sobrepoe a outro. Comercial nao ve acordos. Juridico nao ve leads. Tudo dentro do MdZap, com componentes reutilizaveis mas adaptados por agente.
Cada agente tem:
  • Numero WhatsApp exclusivo
  • Dashboard com KPIs proprios
  • Chat/bate-papo dedicado
  • Painel lateral especifico (lead vs cliente vs processo)
  • Prompt/treinamento juridico proprio
  • Permissoes de acesso por perfil
  • Quick-actions especificas
  • Conexoes com paginas diferentes do MdFlow

🔀 Decisoes Arquiteturais
DECISAO ABERTA ARQ-01
Rotas separadas ou abas dentro do MdZap?
Opcao A — Rotas separadas (/mdzap/leads, /mdzap/docs, /mdzap/acordos-cp)
Cada agente e uma rota independente. Uma rota /mdzap geral serve de hub com acesso a todas.
+ Permissoes naturais por rota (comercial so ve /mdzap/leads) · + Cada rota carrega so seus dados · + URL compartilhavel · + Dashboard/painel totalmente independentes · + Sidebar do MdFlow ja mostra sub-rotas
Opcao B — Abas dentro de /mdzap
Uma unica rota /mdzap com tabs no topo: [ Leads | Docs | Acordos CP | Acordos HM ]
+ Transicao rapida entre agentes · - Carrega tudo junto (pesado) · - Permissoes por aba = logica extra · - Dashboard/painel compartilhados = complexidade · - Mobile: tabs demais
Opcao C — Hibrido (RECOMENDADA CC)
Rotas separadas (/mdzap/leads, /mdzap/docs, etc.) mas compartilhando componentes reutilizaveis (MdzapMsgThread, MdzapMsgInput, etc.). Rota /mdzap como hub de navegacao.
+ Melhor dos dois mundos · + Permissoes nativas · + Componentes DRY · + Cada rota leve · + Hub central para gestores
Impacto da decisao
Define a estrutura de pastas, routing, lazy loading, permissoes e sidebar do MdFlow inteiro. Deve ser decidida ANTES de qualquer codigo.

DECISAO ABERTA ARQ-02
Acordos: 1 conversa por processo ou 1 conversa por escritorio adverso?
Opcao A — 1 conversa por processo
Cada processo gera uma conversa separada com o escritorio adverso. Painel mostra dados daquele processo especifico.
+ Contexto isolado por processo · + Historico claro · - Muitas conversas se o cliente tem N processos
Opcao B — 1 conversa por escritorio/contato
Uma conversa por escritorio adverso, onde se negocia multiplos processos do mesmo cliente.
+ Menos conversas · + Mais natural (telefone real) · - Contexto misturado · - Painel precisa listar N processos

🔄 Fluxo Julia — 9 Steps (Lead ate Assinatura)
A detalhar — Christian vai gravar audios especificos por etapa
Fluxo positivo transcrito. Cada step precisa: mensagens exatas, cenarios negativos, handoff humano/agente.
1
Lead Entra (Inbound via criativo)

Manda nome + CPF. Julia identifica novo contato ou existente.

2
Coleta CPF + Nome Completo

Julia valida CPF, busca no CRM. Se ja existe, retoma contexto.

3
Auditoria Automatica

Dispara auditoria Serasa/CNIS. Julia: "realizando auditoria, aguarde".

4
Analise de Viabilidade

Resultado da auditoria → Julia/humano analisa. Viavel: prossegue. Inviavel: explica.

5
Solicita 2 Documentos + Explica Servico

Julia pede docs iniciais e explica remocao de restricoes. Quais 2: a detalhar.

6
Doc Reader (Sub-agente Extrator)

Processa cada doc recebido. Extrai dados estruturados. Julia coordena espera.

7
CTPS (Carteira de Trabalho)

Humano ou Julia pede. Critico: Julia sabe se humano ja pediu (memoria compartilhada).

8
Procuracao + Contrato

"Cadastro quase pronto, falta assinar." Humano gera procuracoes → envia link.

9
Assinatura + Finalizacao

Lead assina. CRM atualizado. Cliente cadastrado. Processo distribuido. Fim.
