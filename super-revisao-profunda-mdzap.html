1. Visão Geral da Arquitetura

O que é o MDZap

Sistema de mensageria WhatsApp integrado ao CRM MdFlow. Gerencia conversas com leads via WhatsApp usando IA (Julia) como SDR automatizado, com capacidade de handoff para humanos.

5.1 Análise da fn_mdzap_conversas (RPC Principal)

Pontos Positivos
  • ✅ SECURITY DEFINER + SET search_path = public
  • ✅ Paginação server-side com LIMIT/OFFSET (max 100)
  • ✅ Busca com unaccent(lower()) — suporta acentos
  • ✅ Filtragem por agente_slug com COALESCE (fallback sdr_leads)
  • ✅ Filtro deleted_at IS NULL
  • ✅ JOINs com leads, clientes e whatsapp_numeros para dados completos
  • ✅ Retorna envelope JSONB padrão {data, total, page, per_page}
Pontos de Atenção
  • ⚠️ O frontend (useMdzapConversas) extrai (rpcResult as any)?.data — funciona mas perde type safety
  • ⚠️ Não filtra por lead.deleted_at — pode retornar conversas com leads soft-deleted
  • ⚠️ COUNT(*) + SELECT separados (2 queries) — poderia usar window function para 1 query

12. Achados P1/P2

P1-01: RLS genérica em mdzap_julia_lab_sessoes

P1-02: mdzap_conversas_timeline sem deleted_at no RLS
Policy ALL sem filtro deleted_at — retorna registros soft-deleted.
P1-03: Triggers ausentes em tabelas core
Nenhum trigger set_updated_at encontrado em whatsapp_conversas, whatsapp_mensagens ou whatsapp_numeros. Padrão MdFlow exige trigger em toda tabela com updated_at.
P1-04: fn_mdzap_conversas não filtra lead.deleted_at
LEFT JOIN com leads sem filtro deleted_at — pode retornar dados de leads soft-deleted.
P1-05: whatsapp_templates sem RLS
Tabela de templates sem policies de RLS visíveis. Qualquer authenticated pode acessar templates de qualquer org.
P1-06: useMdzapSmartReplies tem fallback hardcoded
Se RPC falha no load inicial, exibe 3 frases hardcoded em português. Deveria exibir estado vazio ou retry.
P1-07: MdzapAppPage.tsx duplica lógica do MdzapChatPanel
280 linhas que replicam boa parte da lógica do ChatPanel (258 linhas). Viola DRY — deveria usar ChatPanel internamente.
P1-08: Hub.tsx usa cores hardcoded em botão favoritar
text-yellow-400 e text-yellow-300 — deveria ser text-warning.
P2-01: useMdzapConversas usa as any em 3 locais
markAsRead e QUERY_KEY dependency — aceito pelo padrão do projeto mas não ideal.
P2-02: mdzap_message_queue sem uso no frontend
Tabela existe no banco mas nenhum componente ou hook a referencia. Possivelmente usado apenas pelo n8n.
P2-03: whatsapp_followups sem uso no frontend
Mesma situação — tabela sem referência no frontend. Reservada para funcionalidade futura.

14. Fluxos Detalhados

14.1 Fluxo de Envio de Mensagem

1. Usuário digita texto e clica Enviar 2. useMdzapSendMessage.sendText() chamado 3. INSERT otimista em whatsapp_mensagens (status: 'pending', remetente_tipo: 'human') 4. invalidateQueries → useMdzapMensagens refetch → msg aparece na thread 5. callWebhook('mdzap/enviar-mensagem') com mensagem_id 6. n8n recebe → envia via WhatsApp API → UPDATE mensagem com wa_message_id + status='sent' 7. Realtime canal mdzap-thread:conversaId detecta UPDATE → debounce 300ms → refetch 8. Thread atualiza: tick ✓ muda para ✓✓ (delivered) e eventualmente ✓✓ azul (read)

14.2 Fluxo de Assumir Conversa (Handoff Julia → Humano)

1. Usuário clica "Assumir" no ThreadHeader 2. useConversaAssumirDevolver.handleAssumir() chamado 3. callWebhook('mdzap/assumir-conversa') com conversa_id + numero_id 4. n8n: marca lead awaiting_human=true, salva assigned_user_id em metadata 5. Resposta: action='toast_refresh' → toast.success + invalidateQueries 6. Realtime canal mdzap-leads-rt detecta UPDATE em leads → invalidate conversas 7. ConversaList: card mostra badge "Humano" (prioridade 5) 8. ThreadHeader: mostra badge "Atendido por [Nome]" + botão "Devolver para Julia"

14.3 Fluxo de Criação de Lead + Auditoria Inline

1. Usuário no LeadPanel → tab Dados → preenche formulário 2. Se lead não existe: botão "Criar Lead & Auditar" 3. useLeadMutations.criarLeadEAuditarInline() chamado 4. INSERT em leads com dados do form + telefone da conversa 5. UPDATE whatsapp_conversas SET lead_id = novo_lead_id 6. RPC rpc_disparar_auditoria (Boa Vista) com CPF do lead 7. Retorna auditoriaId → setLocalAuditoriaId 8. Tab muda para "Restrições" automaticamente 9. LeadRestricoesMdzapTab mostra CrmAuditoriaPanel com polling de status 10. Quando auditoria conclui → lista de restrições aparece

14.4 Fluxo de Pending Lead Data (Doc Reader)

1. Julia recebe documento (CTPS, RG, etc.) do lead no WhatsApp 2. n8n processa via Doc Reader → extrai dados (nome, CPF, etc.) 3. UPDATE whatsapp_conversas SET metadata.pending_lead_data = {...dados...} 4. Realtime canal mdzap-conversas-rt detecta UPDATE 5. useMdzapConversas atualiza cache → conversa.metadata.pending_lead_data populado 6. useLeadFormData detecta pending_lead_data → pré-preenche formulário 7. LeadDadosTab mostra badge "Doc Reader" com campos auto-preenchidos 8. FieldConfidenceBadge mostra confiança (alto/médio/baixo) por campo 9. Usuário revisa → clica "Salvar" ou "Criar Lead"

15. Veredicto Final

Nota: 6.5/10 — Arquitetura excelente, bloqueado por type safety

O MDZap possui uma arquitetura muito bem desenhada: layout responsivo 3 colunas, realtime com fallback polling, scroll infinito, INSERT otimista, Action-Driven UI, 9 níveis de badge status, canal de typing broadcast, e fluxo Doc Reader→Lead automatizado.

O que impede nota 9+: 23 erros de build TypeScript em 3 arquivos críticos (JuliaLab, MsgThread, useLeadFormData). Todos são causados pelo mesmo problema: o types.ts gerado pelo Supabase não reflete corretamente as tabelas mdzap_julia_lab_sessoes, advogados e leads_origens. O fix é trivial (as any nos parâmetros) mas bloqueia o build.

Sem os build errors, a nota seria 8.5/10 (P1s restantes: RLS templates, triggers faltando, DRY violado em AppPage, hardcode yellow no Hub).
