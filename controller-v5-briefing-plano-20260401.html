Controller V5 — Briefing Backend

Contrato completo entre frontend e backend: premissas invioláveis, view principal, RPCs, tipos TypeScript, JSONB analise_ia e wireframe dos cards. Este documento é o "mecanismo" — o frontend deve implementar EXATAMENTE o que aqui está descrito.

9 Premissas Invioláveis 8 RPCs 65+ colunas na view 5 OABs parametrizadas 3 status (não mais 5) Autores: Christian + João Victor + Jarbas 2.0
🔒 9 Premissas Invioláveis Violação = PAUSA imediata na implementação
🚨
Estas 9 premissas são acordos firmados com Christian e João Victor. Qualquer divergência no frontend deve ser identificada como blocker P0 antes de mergear.
P1
Fluxo de status: nao_lida → lida → finalizada

Apenas estes 3 status existem. Os antigos processada, ignorada, arquivada foram abolidos. A migration de banco é obrigatória antes do frontend.

P2
Lida = alteração visual APENAS

Marcar como lida não remove o card da lista nem muda sua posição. Apenas altera a tipografia (font-weight: normal, sem negrito).

P3
Finalizada = line-through + opacity 0.4 (permanece visível)

Card continua na lista com text-decoration: line-through e opacity: 0.4. Só some se o filtro "Ver finalizadas" estiver desligado.

P4
ZERO prazo nos cards da lista

Prazo NUNCA aparece no card da lista de intimações. Aparece apenas: KPI "Vence hoje", drawer aba Intimação, e aba Equipe. Exibir prazo no card = P0 bloqueante.

P5
Urgente + importante coexistem

Uma intimação pode ser simultaneamente urgente (🔴 dot vermelho) e importante (⭐ ring amarelo). São flags independentes — toggle buttons sempre visíveis no card.

P6
Classes IA (A–G) NUNCA exibidas

A classificação interna da Cecília (Classe A, B, C, D, E, F, G) é informação interna de roteamento. Nunca mostrar na UI. KPI "Estratégicas" = classes E+F, mas o label é "Estratégicas" — nunca "Classe E" ou "Classe F".

P7
Mudança de fase via dialog, nunca automática

Avançar fase do processo sempre exige diálogo de confirmação com o advogado. O sistema nunca avança fase automaticamente mesmo que a IA sugira.

P8
Ciência com Renúncia = confirmação humana SEMPRE

Modal especial com 3 checkboxes obrigatórios + campo de observação. Esta ação autoriza peticionamento automático via JARBAS no EPROC. Irreversível — nunca pode ser automática.

P9
5 OABs parametrizadas (não hardcoded)

As 5 OABs vêm do banco (tabela jarbas_accounts). O filtro "Advogado + OAB" é dinâmico. Nunca hardcodar nomes ou números de OAB no frontend.

⚙ 8 RPCs — Contrato de API
⚠
Todas as RPCs são SECURITY DEFINER, retornam JSONB com envelope {success, data, error_code, message}, e verificam organizacao_id internamente. O frontend nunca acessa as tabelas diretamente.
fn_controller_v5_kpis
fn_controller_v5_kpis(p_org uuid, p_advogados text[] DEFAULT NULL)
Retorna os 4 KPIs da barra superior: nao_lidas, urgentes, vence_hoje, estrategicas. Aceita filtro opcional por array de logins de OAB. Executado no carregamento da página e a cada 30s (staleTime).
READ-ONLYSECURITY DEFINER
fn_controller_v5_list
fn_controller_v5_list(p_org uuid, p_filtros jsonb, p_page int DEFAULT 1, p_per_page int DEFAULT 50)
Lista paginada com todos os filtros aplicados no banco. O JSONB p_filtros aceita: status, urgente, importante, advogados[], tribunais[], ciencia_renuncia, requer_atividade, q (busca), sort. Retorna total + items.
READ-ONLYSECURITY DEFINER
fn_controller_marcar_lida
fn_controller_marcar_lida(p_ids uuid[], p_org uuid)
Marca um array de intimações como "lida". Só funciona para status "nao_lida". Registra em eventos_operacionais. Retorna count de atualizados.
BULKSECURITY DEFINER
fn_controller_finalizar
fn_controller_finalizar(p_ids uuid[], p_org uuid, p_notas text DEFAULT NULL)
Finaliza um array de intimações (status → "finalizada"). Funciona para nao_lida e lida. Notas opcionais são salvas no histórico. Revertível via fn_controller_reabrir.
BULKSECURITY DEFINER
fn_controller_toggle_flag
fn_controller_toggle_flag(p_id uuid, p_org uuid, p_field text, p_value boolean)
Toggle genérico para campos booleanos: urgente e importante. p_field aceita "urgente" | "importante". Atualiza e retorna o novo valor.
WRITESECURITY DEFINER
fn_controller_ciencia_renuncia
fn_controller_ciencia_renuncia(p_id uuid, p_org uuid, p_actor_id uuid, p_obs text DEFAULT NULL)
Registra a confirmação de Ciência com Renúncia. Atualiza flags no JSONB, gera evento auditável em eventos_operacionais com actor_id, timestamp e observações. Aciona webhook n8n para peticionamento JARBAS.
P0 — IRREVERSÍVELSECURITY DEFINER
fn_controller_nova_atividade
fn_controller_nova_atividade(p_payload jsonb)
Cria atividade vinculada ao processo. Payload: {intimacao_id, tipo, descricao, prazo, prioridade, responsaveis[], observacoes, organizacao_id, actor_id}. Retorna id da atividade criada.
WRITESECURITY DEFINER
fn_controller_nova_decisao
fn_controller_nova_decisao(p_payload jsonb)
Registra decisão judicial. Payload: {processo_id, resultado, resumo, data_decisao, valor_condenado_centavos, organizacao_id, actor_id}. Resultado: favoravel_total | favoravel_parcial | desfavoravel.
