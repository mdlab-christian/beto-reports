Bugs Mapeados — P0

4 Bugs Críticos Identificados
BUG-01 — P0 CRÍTICO
N8N-REL-CONSULTA Inativo
Sintoma: Valores de restrições retornados incorretamente para todos os leads
Causa provável: Workflow desativado manualmente — provável manutenção incompleta ou deploy com erro
O que investigar: Verificar último log de execução, por que foi desativado, se há erro de credencial ou node quebrado
Solução proposta: Reativar + testar com lead real + adicionar filtro de cartórios como node adicional
Arquivo afetado: n8n → N8N-REL-CONSULTA workflow
Criticidade: MÁXIMA — afeta todos os leads do sistema
BUG-02 — P0 CRÍTICO
Document Received — apenas 3 nodes
Sintoma: Comprovantes enviados pelo WhatsApp não são salvos como Tier 1 no banco
Causa: Workflow simplista (3 nodes) apenas recebe o webhook mas não processa, classifica nem persiste o documento
O que investigar: Quais os 3 nodes existentes e o que cada um faz atualmente
Solução proposta: Expandir para ~15 nodes: receber → identificar tipo → upload Storage → insert processos_documentos (Tier 1, restricao_id=NULL) → notificar atendente
Arquivo afetado: n8n → /MDZAP — Document Received
Criticidade: ALTA — documentos perdidos silenciosamente
BUG-03 — P0 BLOQUEANTE
Sem filtro de cartórios no n8n de auditoria
Sintoma: Restrições de cartórios aparecem como processáveis — não são, pois cartório ≠ credor negociável
Causa: O workflow de auditoria não diferencia entre negativação bancária (processável) e protesto em cartório (não processável automaticamente)
O que investigar: Critérios de exclusão: só por nome? Há lista padrão de cartórios? Inclui CDL e SPC?
Solução proposta: Node de filtro no n8n: se orgao_nome ILIKE '%cartório%' OR '%cartorio%' → set processavel=false, motivo='protesto cartório'
Arquivo afetado: n8n → N8N-REL-CONSULTA + rpc_disparar_auditoria
Criticidade: ALTA — afeta cálculo de processabilidade
BUG-04 — P0 A INVESTIGAR
Histórico inconsistente
Sintoma: Histórico de eventos do lead/cliente mostra dados incorretos ou faltantes
Causa provável: Possível: (1) query sem filtro deleted_at, (2) eventos_operacionais sem RLS correto, (3) timestamp com timezone errado
O que investigar: 1. Schema da tabela de histórico/eventos. 2. Query do MdzapConversaTimeline. 3. Se há overlap entre events de lead vs cliente
Solução proposta: Depende da causa — investigar antes de implementar. Checar query + RLS + join com leads
Arquivo suspeito: MdzapConversaTimeline.tsx + tabela eventos_operacionais
Causa ainda não confirmada — investigar antes de F1

Filtro cartórios: a exclusão como processável é só pelo nome "cartório" ou existe uma lista específica de órgãos?
