O Que Falta — 5 Gaps Identificados

SEÇÃO 3
GAP-01 google_drive_id nunca é preenchido em processos_documentos P0 CRÍTICO
Tabela: processos_documentos
Campo afetado: google_drive_id
Situação: 0% preenchido
Esforço: 30 min

Impacto: Não há rastreamento de qual arquivo no Google Drive corresponde a qual documento do MdFlow. Impossível deletar, mover ou referenciar arquivos Drive de forma programática.

Causa provável: O Code node do n8n faz PATCH usando o campo drive_file_id mas o campo real no banco é google_drive_id. Nome errado → PATCH silencioso sem erro mas sem efeito.

Solução: Abrir o workflow 0QxdwlNqJ1zBy7T6 no n8n → Code node "Salvar pasta + upload arquivos" → localizar a linha do PATCH → corrigir o nome do campo para google_drive_id.

GAP-02 Subpasta de restrição nunca é criada P0 CRÍTICO
Tabela: leads_restricoes
Campos afetados: drive_pasta_id + drive_pasta_url
Situação: 0% preenchido (campos existem)
Esforço: 2–3 horas

Impacto: Todos os documentos Tier 2 (prints, comprovantes de restrição) vão para a pasta raiz do cliente, sem organização por credor. Pasta do cliente vira bagunça com dezenas de arquivos misturados.

Situação: Os campos drive_pasta_id e drive_pasta_url já existem no banco mas nunca foram populados. A infraestrutura está pronta — falta o fluxo que os preenche.

Solução (3 partes):

  • 1. Novo endpoint n8n: mdflow/drive-criar-pasta-restricao
  • 2. Lógica: verificar pasta do cliente → buscar/criar Restrições/ → criar {empresa_nome}/ dentro
  • 3. Frontend: no hook useSalvarRestricao → after success → fire-and-forget novo webhook
GAP-03 Hierarquia de subpastas não existe — pasta flat P1 IMPORTANTE
Componente: WH-SYNC-STORAGE-DRIVE (n8n)
Afeta: toda criação nova de pasta
Esforço: 1–2 horas

Situação atual: Cria pasta flat {Nome} — {id_8char}/ diretamente no root da org.

Desejado: Na criação da pasta do cliente, criar também 3 subpastas fixas: Documentos Pessoais/, Restrições/ e Lixeira/.

Solução: Adicionar 3 chamadas HTTP em sequência no WH-SYNC-STORAGE-DRIVE após criar a pasta principal. Nenhuma mudança de banco necessária para esta fase (os IDs das subpastas fixas não precisam ser salvos — serão buscados por nome quando necessário).

GAP-04 Soft delete não move arquivo para Lixeira no Drive P1 IMPORTANTE
Tabela: processos_documentos
Trigger: soft delete (deleted_at)
Esforço: 2 horas

Impacto: Quando atendente exclui um documento no MdFlow, o arquivo permanece na pasta Drive do cliente sem qualquer indicação de exclusão. Acumula lixo no Drive com o tempo.

Opção A (recomendada): Novo endpoint mdflow/drive-mover-lixeira — recebe google_drive_id + cliente_id → busca/cria pasta Lixeira/ → move arquivo (PATCH parents no Drive).

Opção B: Usar a lixeira nativa do Google Drive via PATCH /drive/v3/files/{id} com trashed: true. Mais simples mas retenção é apenas 30 dias (Google deleta automaticamente).

Requer: Migration para adicionar drive_deleted_at em processos_documentos (ver Seção 8).

GAP-05 Pasta não criada automaticamente ao criar lead P2 DESEJÁVEL
Workflow n8n: MDZAP — Lead Sync
Trigger: criação de lead via MdZap
Esforço: 1 hora

Situação atual: Atendente precisa clicar manualmente no botão "Criar Pasta" para cada novo cliente.

Solução: No workflow MDZAP — Lead Sync → após INSERT do lead → verificar se cliente já tem pasta Drive → se não tem, disparar WH-SYNC-STORAGE-DRIVE com arquivos: [].

Dependência: Requer que GAP-03 (hierarquia) já esteja implementado para criar a estrutura correta desde o início.

Perguntas para Christian — Decisões Pendentes

SEÇÃO 9
Estas decisões impactam a implementação. Precisam de resposta antes de avançar em D2, D3 e D4.
Pergunta 1 — Nomenclatura de pastas
Usar {CPF} — {Nome} ou {Nome} — {id_8char} como nome da pasta do cliente?
CPF no nome da pasta expõe dado sensível (LGPD) para qualquer pessoa com acesso ao Google Drive da org. Recomendação: usar só {Nome} — {id_8char}, que é identificável sem expor dados pessoais. Ou usar {Nome} sem ID caso não haja clientes com nome idêntico.
Pergunta 2 — Credential: OAuth2 vs Service Account
Manter OAuth2 atual (W1vM6djiMMUFil0R) ou migrar para Service Account?
OAuth2 pode expirar (refresh token vence se ficar +6 meses sem uso, ou se a conta Google revogar). Service Account é mais estável para automação headless e não depende de sessão de usuário. Recomendação: migrar para Service Account se houver acesso à Google Cloud Console do MP Advocacia. Menos urgente agora, mas importante no médio prazo.
Pergunta 3 — Root ID da org MP Advocacia
O campo organizacoes.google_drive_root_id está preenchido para MP Advocacia?
Sem o root_id, o n8n não sabe onde criar as pastas. Verificar no Supabase: SELECT nome, google_drive_root_id FROM organizacoes WHERE nome ILIKE '%midas%' OR nome ILIKE '%mp%'. Se vazio, precisamos definir qual pasta do Drive será a raiz e cadastrar o ID.
Pergunta 4 — Bidirecional Drive → MdFlow
Confirmar que F6 (Drive → MdFlow) está fora do escopo por agora?
A implementação bidirecional exigiria Google Drive push notifications (watch API), um endpoint público para receber os eventos e lógica de dedup/mapeamento de arquivos. Complexidade alta, valor baixo a curto prazo. Recomendação: deixar para uma fase futura quando o fluxo MdFlow → Drive estiver maduro e estável.
Pergunta 5 — Lixeira: pasta manual vs lixeira nativa Google
Preferir pasta Lixeira/ manual dentro da pasta do cliente, ou usar a lixeira nativa do Google Drive?
Opção A — pasta Lixeira/ manual: arquivo continua visível no Drive em uma subpasta, fácil de recuperar, sem prazo de expiração automático.

Opção B — lixeira nativa Google: mais simples de implementar (só setar trashed: true), mas Google deleta automaticamente após 30 dias. Arquivo some do Drive sem ação manual.

Recomendação: Opção A (pasta Lixeira/) — mais controle, sem risco de perda automática de arquivos jurídicos.
