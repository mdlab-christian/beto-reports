P0 CRÍTICO — service_role Key Hardcoded

A chave service_role do Supabase (eyJhbGci...) está hardcoded como string literal em pelo menos 10 code nodes distribuídos por 7 workflows ativos. Qualquer export de workflow ou acesso ao n8n exporia a chave que dá acesso total ao banco de produção, bypassando RLS.

Workflow Node(s) Afetado(s) Credencial
Inbound Upsert Lead Process SUPABASE_KEY + N8N_SECRET
Enviar Mensagem FE Parse and Fetch Config + Save Saida Supabase SUPABASE_KEY (2x)
Arquivar Conversa Arquivar Conversa SUPABASE_KEY
Devolver Julia (novo) Process SUPABASE_KEY
MDZAP-DEVOLVER-JULIA (legado) Code SUPABASE_KEY
Julia Lab Refine Refine Prompt SUPABASE_KEY + OPENAI_KEY
Julia Prompt Save 3x httpRequest nodes SUPABASE_KEY (3x)
Message Grouper Process Queue SUPABASE_KEY
Audio TTS Process SUPABASE_KEY + ELEVENLABS_KEY
FOLLOWUP-ZAPSIGN Buscar + Processar (2 nodes) SUPABASE_KEY (2x)

Fix: Configurar SUPABASE_KEY, OPENAI_KEY, ELEVENLABS_KEY como Environment Variables no n8n (Settings → Variables). Substituir todas as referências por process.env.SUPABASE_KEY nos Code nodes e {{ $env.SUPABASE_KEY }} nos httpRequest nodes. Esforço: ~30min para todos os 10 workflows.

P0 CRÍTICO — Webhook mdzap/assumir-conversa Inexistente

O frontend chama callWebhook('mdzap/assumir-conversa', ...) em dois hooks:

  • useLeadPanelActions.ts (handleAssumir)
  • useMdzapThreadActions.ts (handleAssumir)

Não existe nenhum workflow ativo ou inativo com esse path no n8n.

O botão "Assumir" no MdZap está fazendo POST para um endpoint que retorna 404 (ou 500 dependendo da config Railway).

Fix: Criar workflow /MDZAP — Assumir Conversa com lógica: UPDATE whatsapp_conversas.metadata.mode='human', UPDATE leads.awaiting_human=true, INSERT timeline + leads_atividades. Espelhar o inverso do "Devolver Julia".

MDZAP-DEVOLVER-JULIA (legado)

Fix: Desativar o legado (CGPoGVlUnr28xI52). Adicionar headerAuth ao novo. Mesclar o WH log do legado no novo.

P0 CRÍTICO — Julia Prompt Save Sem Autenticação

O workflow /MDZAP — Julia Prompt Save (6jgyFxNBWVrkI132) está com o webhook completamente aberto — sem headerAuth, sem validação inline.

Qualquer POST para /webhook/mdzap/julia-prompt-save pode criar/modificar prompts da Julia em qualquer organização.

Fix: Adicionar authentication: headerAuth no node Webhook. 1 clique no n8n.

Race Conditions Encontradas

Julia Lab Refine

1. Desativa prompt atual (PATCH ativo=false)
2. Insere novo prompt (INSERT)
Se INSERT falhar → fica sem prompt ativo

Fix: Inverter: INSERT primeiro, depois PATCH. Ou usar transação via RPC.

Julia Prompt Save

1. Busca max versão (GET)
2. Desativa atuais (PATCH) ← paralelo!
3. Calcula nova versão
4. Insere nova versão
2 requests simultâneos → versões duplicadas

Fix: Usar RPC atômico fn_save_julia_prompt que faz tudo em 1 transação.

Veredicto Final

3.5/10 — O ecossistema n8n do MdZap é a parte mais frágil de todo o módulo. A nota é puxada para baixo por problemas de segurança indefensáveis (service_role hardcoded em 7 workflows) e funcionalidades core ausentes (assumir conversa sequer existe, memória e doc-reader nunca foram criados). A presença de 4 stubs vazios retornando success: true sem fazer nada é particularmente insidiosa — o frontend acredita que funcionou. Os 2 workflows duplicados no mesmo path criam comportamento imprevisível em produção.

Pontos positivos: Message Grouper com lógica anti-storm (janela 8s) é inteligente. FOLLOWUP-ZAPSIGN é o workflow mais completo e bem documentado. O envelope canônico é respeitado pelo frontend. Z-API: limpo nos ativos. O caminho para 6.0 exige: mover keys para env vars (30min), criar Assumir Conversa (1h), desativar legado (1min), adicionar auth (5min), reativar 3 workflows (1.5h), deletar dead code (15min). Total: ~3.5h para quase dobrar a nota.
