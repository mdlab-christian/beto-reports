§02 — Sistema SDR — 3 Agentes

agente_slug driven
ℹ️
Cada instância WhatsApp tem um agente_slug que determina TUDO: prompt, comportamento, pipeline CRM, visual no frontend. O roteamento é 100% dinâmico — zero hardcode de número ou agente.
Julia

Cria lead → INSERT leads com coluna_crm = 'novos'

Acordo confirmado → gera link ZapSign via WH-16
Atualiza acordos.status na tabela financeira
Lógica de Roteamento por phone_number_id
Meta Cloud API envia mensagem └─► phone_number_id no payload (metadata.phone_number_id) └─► SELECT agente_slug FROM whatsapp_numeros WHERE wa_instance_id = phone_number_id AND deleted_at IS NULL └─► agente_slug decide TUDO: ├─ qual workflow n8n executar ├─ qual system_prompt usar (mdzap_ai_modules) ├─ qual comportamento especial ativar └─ qual visual mostrar no frontend
Fallback de Prompt (igual sistema legado)
1° Busca mdzap_ai_modules WHERE slug = agente_slug AND organizacao_id = $orgId
2° Fallback: slug = agente_slug AND organizacao_id IS NULL (global)
3° Último recurso: prompt padrão genérico hardcoded
Variáveis de Template nos Prompts
{{contact_name}} nome do contato WA
{{cpf}} CPF do lead (se informado)
{{coluna_crm}} estágio atual no CRM
{{turn_count}} número do turno da conversa
{{data_hora}} data e hora atual (BRT)
{{nome_empresa}} nome da organização
