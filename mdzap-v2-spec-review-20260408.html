Seção 1 — Fluxo de Atendimento Julia (6 Fases)

Contexto: Este fluxo descreve a lógica que o frontend deve suportar. O prompt da Julia em si é responsabilidade do Matheus — não está nesta iteração.
F1
Fase 1 — Entrada do Lead

Lead clica no anúncio Facebook Ads → WhatsApp abre com nome + CPF pré-preenchidos.

Julia
  • Recebe mensagem com nome + CPF
  • Cadastra contato automaticamente (nome + número)
  • Envia boas-vindas apresentando "Midas Restrições"
  • Explica: auditoria de crédito, descobrir brechas
  • Pergunta: tem interesse em descobrir a situação no CPF?
Regras desta fase
  • NUNCA citar nome do escritório de advocacia
  • NUNCA citar nome de advogado
  • Empresa apresentada: "Midas Restrições" — apenas
F2
Fase 2 — Coleta de Documentos

Julia pede CNH/RG + comprovante de residência. Lead envia em qualquer ordem.

Julia
  • Solicita documento de identificação (CNH ou RG)
  • Solicita comprovante de residência
  • Se comprovante não bate com titular → pede declaração de terceiro
  • NUNCA sugere extrato bancário
  • Sugere boleto de banco digital (mostra endereço no nome)
Doc Reader — Identificação
  • Verifica legibilidade (ilegível → pede foto melhor/PDF)
  • Extrai: nome, CPF, RG, data nascimento, sexo, validade
  • Salva PDF no storage
  • Score de confiança por campo
Doc Reader — Comprovante
  • Verifica legibilidade
  • Verifica se nome/CPF bate com titular
  • Se bate → extrai endereço → salva PDF
  • Se não bate → passa output para Julia pedir declaração
  • Declaração recebida → processa → extrai endereço
Declaração de terceiro (quando necessário)
  • "Eu, [nome terceiro], CPF [X], declaro que [nome lead], CPF [Y], reside comigo nesta moradia."
  • Documento de identificação do terceiro obrigatório
F3
Fase 3 — Pré-cadastro (pending_lead_data)

Doc Reader popula o metadata da conversa. Atendente confirma e cria o lead formal.

Doc Reader
  • Popula whatsapp_conversas.metadata.pending_lead_data
Atendente (Frontend)
  • Vê formulário pré-preenchido no MdZap
  • Confirma os dados
  • Cria lead formal com coluna CRM: em_atendimento
Decisão reunião (P2)
  • Opção B confirmada: ao criar o lead, coluna CRM = 'em_atendimento'
  • NÃO fazer cadastro automático em massa — risco de overload
F4
Fase 4 — Auditoria (Atendente Assume)

Notificação aparece no MdZap → atendente assume → faz auditoria Boa Vista → sistema cadastra restrições.

Atendente
  • Assume conversa via notificação de auditoria pendente
  • Faz auditoria Boa Vista pelo sistema MdFlow
  • Sistema cadastra restrições automaticamente
  • Comprovante PDF gerado pelo próprio MdFlow (não API externa)
Julia (pós-auditoria)
  • [Nome limpo] → parabeniza → encerra conversa
  • [Nome sujo] → informa SOMENTE empresas (nunca valores)
  • "Vimos X ocorrências da empresa [Y]"
  • Prazo máximo: "90 dias para o primeiro resultado"
  • Indique Ganho → FEATURE FUTURA (não mencionar)
Regras desta fase
  • Só Boa Vista por enquanto — Serasa não integrado
  • NUNCA mencionar valores das restrições, apenas empresas
  • NUNCA dar garantia — prazo máximo "90 dias para primeiro resultado"
F5
Fase 5 — CTPS / Elegibilidade AJG

Julia pede CTPS para comprovar elegibilidade AJG. Teto varia por estado.

Julia
  • Pede CTPS — explica o porquê: comprovar renda abaixo do teto para AJG
  • NUNCA usa o termo "JG" — usa sempre "AJG"
  • RS: teto = 5 salários mínimos
  • SC e SP: teto = 3 salários mínimos
  • Se autônomo → nós fazemos renda/receita (não Julia)
  • Se desempregado → cadastra profissao = "Desempregado"
Doc Reader — CTPS
  • [Contrato aberto] → extrai cargo (= profissão) + data_admissao → UPDATE leads.profissao
  • [Contrato fechado] → extrai último vínculo → profissão → Julia pergunta se ainda trabalha
  • [Renda receita] → feito por humano → Julia monitora se falta declaração IR
Decisão reunião (P4)
  • cargo = profissão — campo único, não criar campo separado
F6
Fase 6 — Procuração ZapSign

Humano gera procuração via MdFlow → envia link ZapSign → Julia acompanha assinatura.

Humano (atendente)
  • Gera procuração via MdFlow
  • Envia link ZapSign para o lead
  • Lead vai para coluna CRM: aguardando_assinatura
Julia (follow-up)
  • Fica responsável por cobrar assinatura
  • 1h sem resposta → follow-up automático (n8n cron)
  • Lead assina → Julia confirma encaminhamento ao escritório parceiro
  • Lead diz "não quer" → pergunta sobre dúvida → sem resposta → cancela follow-up
Regras desta fase
  • Geração de procuração = humano — Julia NUNCA gera
  • Julia NUNCA cita o nome do escritório ao confirmar encaminhamento
  • Follow-up: máximo X tentativas configurável
