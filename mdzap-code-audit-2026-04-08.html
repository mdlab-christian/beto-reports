4.8/10
Nota Código MdZap

Código funcional mas tecnicamente frágil. A página MdZap é a mais complexa do MdFlow — 4.200+ linhas em 22 arquivos — e acumula dívida técnica severa. O hook central (useLeadPanelActions) tem 8 bugs catalogados. Há 302 linhas duplicadas que reimplementam um componente CRM existente. O sistema de tipos é minado por as any em pontos críticos. A estratégia de query invalidation é inconsistente. O bureau de consulta difere entre MdZap e CRM — um bug de integração que afeta dados reais em produção. Pontos positivos: 3 de 5 tabs do lead panel reutilizam componentes CRM corretamente, o edge function está sólido, e o MsgBubble é competente no render de mídia.

Bureau Mismatch — Bug Crítico Cross-Module

MdZap (useLeadPanelActions)

// saveLead mutation
rpc('fn_consultar_restricoes', {
  p_cpf: cpf,
  p_bureaus: ['boavista']  // <-- ERRADO
})

src/components/mdzap/lead-panel/useLeadPanelActions.ts

CRM (CrmAuditoriaPanel)

// handleConsultar
rpc('fn_consultar_restricoes', {
  p_cpf: cpf,
  p_bureaus: ['apifull_115']  // <-- CORRETO
})

src/components/crm/CrmAuditoriaPanel.tsx

Consequência: O MdZap consulta restrições via bureau boavista (antigo/legado?), enquanto o CRM usa apifull_115 (atual). Se os bureaus retornam dados diferentes, leads cadastrados pelo MdZap terão restrições divergentes do que o CRM mostraria para o mesmo CPF.

Fix: Alinhar para apifull_115 em ambos, ou extrair constante DEFAULT_BUREAUS compartilhada.

Query Management — O Caos Silencioso

Mapa de Queries e Invalidation

Query Key Onde Definida Invalidada Por Problema
['lead', leadId] useLeadPanelActions saveLead (['leads']), criarLead (['leads']) MISMATCH Invalida ['leads'] mas query é ['lead', id]. Funciona por partial match, mas frágil.
['lead-docs', leadId] useLeadPanelActions handleUploadDoc (['lead-docs']) OK Partial match funciona.
['mdzap-conversas'] useMdzapConversas Nada explicitamente. Realtime channel faz invalidation. FRAGILE Depende 100% do Realtime. Se canal cair, stale data.
['restricoes', leadId] useRestricoes (shared) LeadRestricoesMdzapTab (['restricoes']) BROAD Invalida TODAS restricoes de TODOS leads. Funciona mas causa refetch desnecessário.
docMensagens useLeadPanelActions (useEffect) Não é React Query BUG useState + useEffect + .then(). Sem cache, sem retry, sem invalidation.
advogados useLeadPanelActions (useEffect) Não é React Query BUG Mesmo problema. Fetcha em cada render.

Realtime: 3 Channels, 3 Estratégias

useMdzapConversas abre 3 canais Realtime com estratégias mistas:

  • Canal 1 (conversas): INSERT/UPDATE → invalidate queries ✓
  • Canal 2 (mensagens): INSERT → invalidate queries ✓
  • Canal 3 (leads): UPDATE → invalidate queries ✓

Mas nenhum canal filtra por organizacao_id. Em multi-tenant, isso significa que mudanças de QUALQUER organização triggerão invalidation. Não é um security issue (RLS protege o SELECT), mas causa refetch desnecessário em orgs com muitas conversas.

Fallback: Polling de 8s como safety net. Correto mas esconde bugs do Realtime.

Componentes UI — Análise de Qualidade

MdzapMsgBubble (495 linhas) — O Maior Componente

Render de mensagens com suporte a 7+ tipos de mídia. Bem implementado para a complexidade:

Acertos

  • Lightbox para imagens
  • Audio player customizado
  • Retry para mídia falhada
  • Swipe-to-reply gesture
  • Feedback buttons (Julia)
  • Timestamp formatting correto

Problemas

  • 495 linhas — excede limite 500 por pouco, mas deveria ser <400
  • Switch/case com 7+ tipos sem extraição em sub-componentes
  • Re-renders: novo render em cada msg quando scroll muda
  • Sem React.memo — lista de 100+ msgs = performance issue

MdzapConversaList (410 linhas) — Filtragem Problemática

Problema central: Toda filtragem é client-side.

  • SDR mode, status, tipo de atendimento, unread — todos filtrados após fetch completo
  • Busca por nome/telefone: também client-side com debounce
  • Bulk actions: mistas (arquivar/transferir via webhook, mark-unread via direct supabase)
  • Com 8 conversas funciona. Com 800, vai travar.

Fix: Mover filtros para fn_mdzap_conversas RPC. O RPC já aceita parâmetros — basta usá-los.

Plano de Ação — Priorizado

P0 Fix Imediato (bloqueia qualidade de dados)

  1. Bureau mismatch: Alinhar useLeadPanelActions para usar apifull_115 (ou constante compartilhada). 1 linha de código, impacto enorme.
  2. Unwrapped queries: Migrar docMensagens e advogados de useEffect+.then() para useQuery. ~40 linhas de refactor.

P1 Refactor Urgente (dívida técnica ativa)

  1. LeadDocumentosTab: Substituir reimplementação por reuso de ClienteDrawerTabDocumentos com prop origem. Remove 280+ linhas, ganha Drive sync e realtime de graça.
  2. Query invalidation: Padronizar keys. Usar queryClient.invalidateQueries({ queryKey: ['lead', leadId] }) exato em vez de broad ['leads'].
  3. Type safety: Eliminar as any em MdzapMsgThread (L98, L114-115) e useMdzapConversas. Tipar retorno do RPC.
  4. criarLead invalidation: Adicionar invalidateQueries(['mdzap-conversas']) no onSuccess.

P2 Melhoria (qualidade de código)

  1. MsgBubble: Extrair sub-componentes por tipo de mídia + React.memo.
  2. ConversaList: Mover filtros para RPC. Preparar para 800+ conversas.
  3. Realtime channels: Adicionar filtro organizacao_id para evitar invalidation cross-org.
  4. Error handling: Trocar toasts genéricos por error com context (error.message).
  5. PendingLeadData: Usar o tipo ou remover. Dead types poluem.

Veredicto Final

4.8/10 — Código que funciona apesar de si mesmo. O MdZap é a página mais complexa do MdFlow e a que menos respeita os próprios padrões do projeto. A promessa de reuso CRM está 60% cumprida — as tabs fáceis (Notas, Memória, Restrições) foram delegadas corretamente, mas a mais importante (Documentos) foi reimplementada com menos features. O hook central tem 8 bugs catalogados e queries fora do React Query. O bureau mismatch é um bug de dados reais em produção que precisa de fix imediato.

O edge function é o ponto alto (7.0). Se o mesmo rigor fosse aplicado ao código frontend, a nota seria 7+. O caminho para 7.0 exige: fix bureau (5min), migrar 2 queries para useQuery (1h), substituir LeadDocumentosTab por reuso CRM (2h), eliminar as any (1h). Total: ~4h de trabalho para +2.2 pontos.
