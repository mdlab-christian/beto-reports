3. Diagnostico do Page-Loop v1 — Por Que as Paginas Saem com Bugs

12 Gaps Criticos Identificados

  1. Sem pre-flight — nao verifica git limpo, server rodando, spec existente, Supabase conectado. Loop pode falhar apos 2h.
  2. TypeScript nao obrigatorio — page-qa-tester e page-deep-explorer NAO rodam tsc --noEmit. Type drift = data loss silencioso.
  3. Silent-drop audit incompleto — frontend envia param X que RPC nao aceita. Dado desaparece sem erro.
  4. Multi-tenant NAO testado — nenhuma skill testa org_A acessando dados de org_B.
  5. Async webhook fraco — "Gerando..." pode persistir forever sem detecao de falha.
  6. Performance nao testada — paginacao com 100K rows, memory leaks, N+1 queries.
  7. Accessibility nao sistematica — aria-labels verificados parcialmente, Tab order nunca.
  8. Spec drift detection fraco — spec diz campo obrigatorio, codigo faz opcional.
  9. Loops podem bloat — max 3 ciclos × 2 skills × 30min = +3h extras.
  10. Report existence nao validada — page-finalize procede com reports faltando.
  11. Redundancias — spec lido 4x, code audit 4x, visual audit 3x, dirty state 3x.
  12. Componentes compartilhados — fix em ClienteDrawer pode quebrar 3 paginas sem cross-check real.

4. Page-Loop v2 — Arquitetura Redesenhada

Objetivo: rodar page-loop em qualquer pagina e ela sair 100% pronta para uso. Zero bugs bloqueantes. Nota >= 9.5/10.

0
Pre-Flight (NOVO)
~2 min
NOVO — Validacao antes de comecar:
  • Git status limpo (sem uncommitted changes)
  • Dev server rodando (curl localhost:8080 → 200)
  • Supabase MCP conectado (test query → OK)
  • Spec da pagina existe e e valido (docs/_resumido___PAGINA.txt)
  • Reports dir existe (~/jarbas-reports/beto/PAGINA/)
  • TypeScript compila (npx tsc --noEmit → 0 errors) — se falhar, corrige ANTES
  • Conta de teste logada no browser (agent-browser → auth OK)

FAIL FAST: Se qualquer check falhar, PARA. Nao desperdicar 4h pra falhar na fase 3.

1
Funcional + Backend
~60 min
MELHORIA — Fundir QA funcional + Backend audit numa unica fase:

Antes: QA-tester → Executor → (depois) Backend-tester → Backend-executor
Agora: Backend-tester PRIMEIRO → Backend-executor → QA-tester → Executor

Por que: 40% dos bugs que o QA encontra sao causados por schema drift. Corrigir backend primeiro elimina bugs na raiz.

FASE 1A — Backend Audit (page-backend-tester melhorado) ├── Schema audit: colunas, tipos, constraints, indices ├── RPC audit: SECURITY DEFINER, silent-drop, type-drift ├── NOVO: Multi-tenant test (org_A query org_B → 0 rows) ├── NOVO: Soft-delete enforcement (deleted_at IS NULL em TODA query) ├── NOVO: Spec-to-schema validation (campo no spec existe no DB?) ├── Webhook envelope validation └── Gera: backend-test-v{N}.html FASE 1B — Backend Fix (page-backend-executor) ├── Aplica migrations P0→P1→P2 ├── Valida cada migration ├── NOVO: TypeScript check apos cada migration └── Max 2 ciclos (nao 3) FASE 1C — QA Funcional (page-qa-tester melhorado) ├── 10 grupos (A-J) + browser automation ├── NOVO: TypeScript check obrigatorio ├── NOVO: Componentes compartilhados cross-check │ (se ClienteDrawer e usado, testar nos 3 contextos) ├── NOVO: Documento tiers validation │ (Tier 1/2/3 corretos por contexto?) └── Gera: qa-report-v{N}.html FASE 1D — Bug Fix (page-executor) ├── Fix P0→P1→P2 (1 commit/bug) ├── NOVO: Silent-drop check antes de cada fix RPC ├── NOVO: Cross-page impact check │ (se editou ClienteDrawer, verificar /crm + /clientes + /distribuicao) └── Gera: execution-log-v{N}.html
2
Profundidade + UI/UX (Fundido)
~60 min
MELHORIA — Fundir deep-explorer + page-front numa unica skill hibrida:

Antes: Deep-explorer → Polisher → Page-front → Front-executor (4 skills, muita redundancia)
Agora: Deep-front-audit (1 skill hibrida) → Deep-front-executor (1 executor unificado)

Por que: Elimina 3 redundancias (visual audit, dirty state, responsividade). Economiza ~40min e ~50K tokens.

FASE 2A — Deep + Front Audit Unificado (NOVO: page-deep-front-audit) ├── Cria mock data [AUDIT] no browser ├── 6 breakpoints screenshots (375, 640, 768, 1024, 1280, 1440) ├── 13 categorias do deep-explorer (nav, tabs, filters, table, cards, drawers, forms, upload, DnD, buttons, states, responsive, reuse) ├── 12 code checks do page-front (hardcodes, icons, fonts, z-index, glass, tooltips, hover, toasts, dirty, empty, staleTime, file size) ├── 9 notas por categoria (hierarchy, DS, cards, drawers, buttons, typography, states, responsive, forms) ├── NOVO: Documento tiers (Inicial vs Tramitacao) verificacao explicita ├── NOVO: Restricoes tab cross-context check ├── NOVO: A11Y basico (aria-labels, Tab order, focus visible) ├── Cleanup [AUDIT] data └── Gera: deep-front-audit-v{N}.html (UNICO report, nao 2) FASE 2B — Fix Unificado (NOVO: page-deep-front-executor) ├── Prioridade: FUNC → STATE → FORM → RESP → CSS → A11Y ├── CSS-only → JSX → Hook (nao skip automatico) ├── NOVO: Cross-page check se componente compartilhado ├── Loop max 2 ciclos ate nota >= 9.0 └── Gera: deep-front-exec-v{N}.html

5. Checklist Universal — O Que o Page-Loop v2 DEVE Verificar em TODA Pagina

[BK] Backend (10 checks)

  • organizacao_id em TODA query
  • deleted_at IS NULL em TODA query
  • RLS habilitado + policy correta
  • RPCs com SECURITY DEFINER
  • SET search_path = public
  • TEXT CHECK (nunca ENUM nativo)
  • BIGINT centavos (nunca FLOAT)
  • TIMESTAMPTZ (nunca DATE)
  • Trigger updated_at
  • Indices: org_id, org_id WHERE deleted_at IS NULL

[SEC] Seguranca (10 checks)

  • Multi-tenant: org_A nao ve org_B
  • Soft delete: registros deletados nao aparecem
  • Silent-drop: todo param enviado e aceito
  • Type-drift: tipo do param = tipo da coluna
  • Secrets em .env (zero hardcode)
  • anon role nao acessa dados protegidos
  • Mascaramento financeiro (acesso_financeiro)
  • CPF redacted em logs
  • Webhook auth header presente
  • CORS correto

[FUNC] Funcional (10 checks)

  • CRUD completo funciona
  • Filtros aplicam corretamente
  • Busca com unaccent funciona
  • Paginacao server-side
  • Empty states em toda lista
  • Loading states (skeleton, nao spinner)
  • Error states com mensagem
  • Toast apos toda acao
  • Dirty state em modais/drawers
  • Realtime funciona (INSERT/UPDATE/DELETE)

[UX] UI/UX (10 checks)

  • Zero hardcode hex (so tokens CSS)
  • Fonts: Inter body, Manrope H1/H2, JetBrains Mono CNJ/CPF
  • Responsivo: 375, 768, 1024, 1440px
  • Hover states em elementos interativos
  • Tooltip em botoes icone
  • Badges com tokens (nao Tailwind named colors)
  • Glassmorphism so em cards/modais/popovers
  • Z-index correto (modal=50, toast=60, tooltip=70)
  • Truncate com tooltip em texto longo
  • Touch targets >= 44px mobile

[PERF] Performance (10 checks)

  • staleTime: 30_000 em todo useQuery
  • Sem JSON.stringify em deps arrays
  • Sem key={index} em listas
  • select() especifico (nao select('*'))
  • Sem console.log em producao
  • Arquivos < 500 linhas
  • Sem import * as
  • Sem transition-all (usar especifico)
  • Paginacao limit(500) em queries grandes
  • Realtime com filtro organizacao_id

[A11Y] Acessibilidade (6 checks)

  • aria-label em todo botao icone
  • htmlFor em todo label de form
  • Sem outline-none sem alternativa focus
  • role=tablist/tab em tabs
  • Esc fecha modal/drawer
  • Focus auto no primeiro campo do modal

[CODE] Qualidade (6 checks)

  • Zero TODO/FIXME
  • Zero any TypeScript
  • Zero console.log/warn
  • Zero hardcode URL/secret
  • Zero button-in-button
  • Imports limpos (sem dead imports)

Total: 72 checks universais — aplicados em TODA pagina, TODA vez. Nenhum check e "opcional".
