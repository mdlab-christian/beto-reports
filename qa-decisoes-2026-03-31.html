Correcao Banco — Conflito arquivada vs status_analise

O problema

Dois mecanismos faziam a mesma coisa:

  • arquivada (BOOLEAN) — usado pelo frontend em 12+ lugares para filtrar decisoes ativas vs arquivadas
  • status_analise = 'arquivada' — usado por um trigger no banco

O trigger fn_trg_decisao_update_processo filtrava com status_analise <> 'arquivada', mas o indice filtrava com arquivada = false. Resultado: inconsistencia potencial nos dados agregados.

Decisao tomada

arquivada (boolean) e o SSOT para estado de arquivo.

Razao: arquivamento e ortogonal ao status de analise. Uma decisao "aprovada" pode ser arquivada sem perder o status original. Quando desarquivada, o status_analise original e preservado.

  • status_analise: pendente → processando → aprovado / rejeitado / erro (ciclo de analise IA)
  • arquivada: true/false (estado de arquivo, independente)
Migration 20260401000002 — fix_decisoes_arquivada_ssot

1. Trigger corrigido

Agora usa arquivada = false AND deleted_at IS NULL

2. CHECK constraint

status_analise so aceita: pendente, processando, aprovado, rejeitado, erro. Valor 'arquivada' proibido.

3. Dados legados limpos

Registros com status_analise='arquivada' migrados para 'aprovado'.

4. Indices criados

idx_decisoes_org_active e idx_decisoes_org_archived

FK intimacao_id — Pendente de Decisao

Contexto

A tabela decisoes tem um campo intimacao_id (FK para intimacoes), e a tabela intimacoes tem um campo decisao_id (FK para decisoes). Isso e uma FK bidirecional — anti-padrao no MdFlow.

8.693
Decisoes ativas

ZERO com intimacao_id preenchido

561
Intimacoes

ZERO com decisao_id preenchido

decisoes.intimacao_id

"esta decisao foi gerada a partir desta intimacao" — direcao natural (intimacao gera decisao)

intimacoes.decisao_id

"esta intimacao resultou nesta decisao" — mesma informacao, lado inverso. Redundante.

Recomendacao

  • Manter decisoes.intimacao_id — lado canonico
  • Remover intimacoes.decisao_id — redundante, cria dependencia circular
  • Pergunta pendente: Uma intimacao pode gerar mais de 1 decisao?
    • Se NAO (1:1): adicionar UNIQUE(intimacao_id) em decisoes
    • Se SIM (1:N): manter sem UNIQUE, usar apenas o indice parcial existente
