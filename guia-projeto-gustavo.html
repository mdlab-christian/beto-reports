O que é e por que usar

O /projeto é uma skill do Beto CC que te força a planejar antes de implementar. Em vez de pedir pro Beto "faz a tela de audiências" e receber código aleatório, você passa por 4 fases estruturadas.

O resultado é código que segue as convenções do MdFlow, sem improviso, sem TODO, sem placeholder — exatamente o que foi especificado.

📂
Onde ficam os arquivos
Todos os projetos ficam em ~/beto-vault/projetos/<nome>/ — visíveis no seu Obsidian. Você consegue acompanhar o progresso, ler specs e planos direto no vault.
# Sintaxe geral /projeto <nome> <comando> [args] # Exemplos reais /projeto audiencias spec "nova tela de audiências com modal de criação" /projeto audiencias break /projeto audiencias plan T01 /projeto audiencias execute T01 /projeto audiencias next
📋

Fase 1 — spec

1
Criar a especificação
Define o que será construído — absorve CLAUDE.md, docs existentes, schema real do Supabase
spec
/projeto audiencias spec "nova tela de audiências com modal de criação, edição e cancelamento"

O que o Beto faz:

1
Lê o CLAUDE.md do MdFlow
Absorve arquitetura, Design System, convenções de código, regras de autonomia
2
Verifica o schema real no Supabase
Confere tabelas, colunas, RPCs existentes antes de planejar qualquer coisa
3
Lê docs existentes da feature
Se tem spec em ~/beta-mdflow/docs/, absorve em vez de reescrever
4
Gera spec.md completo
Com objetivo, escopo, componentes, comportamentos, backend, automações
⚠️
Revise antes de continuar
Depois do spec, leia o arquivo gerado. Se algo estiver errado aqui, vai errar em todas as fases seguintes. Corrija antes do break.

Onde fica salvo:

~/beto-vault/projetos/audiencias/ ├── spec.md ← gerado aqui └── projeto.json ← metadados do projeto
✅
Faça /clear depois
Após revisar o spec, rode /clear no terminal do Beto antes de passar para o break. Context window limpo = output melhor.
🔀

Fase 2 — break

2
Quebrar em tarefas atômicas
Transforma a spec em tarefas pequenas, por camada, na ordem certa de implementação
break
/projeto audiencias break

Ordem obrigatória de camadas:

Camada O que inclui Por que primeiro
backend Migrations, RPCs, RLS, triggers, índices Banco é lei. Frontend só existe se o backend está certo
frontend Pages, componentes, hooks, types Depende do banco estar pronto
n8n Webhooks, cenários, automações Orquestração depois da estrutura
polish Loading states, empty states, responsividade, a11y Último porque depende de tudo

Regra de tamanho:

S 1–2 arquivos — pode ir direto
M 3–5 arquivos — plan + execute
L 5+ arquivos — dividir primeiro

Exemplo de saída:

## Backend [ ] T01 — Migration tabela audiencias (S) [ ] T02 — RPC get_audiencias_por_processo (S) — Depende: T01 [ ] T03 — RPC criar_audiencia (S) — Depende: T01 ## Frontend [ ] T04 — Página /audiencias layout + tabela (M) — Depende: T01, T02 [ ] T05 — Modal criar/editar audiência (M) — Depende: T03, T04 ## n8n [ ] T06 — Webhook notificação nova audiência (M) — Depende: T01 ## Polish [ ] T07 — Loading/empty states + mobile (S) — Depende: T04, T05

Onde fica salvo:

~/beto-vault/projetos/audiencias/ ├── tasks.md ← legível no Obsidian └── tasks.json ← usado pelo Beto nas fases seguintes
🗺️

Fase 3 — plan

3
Planejar UMA tarefa
Pesquisa no codebase, verifica banco, roda revisão Codex — antes de escrever 1 linha de código
plan <ID>
/projeto audiencias plan T04

O que o Beto faz:

1
Verifica dependências
T04 depende de T01 e T02. Se não estiverem concluídas, avisa antes de continuar
2
Pesquisa no codebase
Acha componentes similares, hooks reutilizáveis, patterns que já existem no projeto
3
Roda Codex Pre-Review (o4-mini)
Revisa os arquivos que serão modificados antes de tocar em qualquer coisa. Detecta any implícito, hex hardcoded, console.log, arquivos >500 linhas, queries sem organizacao_id
4
Gera plano passo a passo
Lista exatamente: qual arquivo, que ação, o que fazer — sem espaço para improviso
🔍
Codex Pre-Review
Antes de implementar, o Beto roda uma revisão automática com o4-mini nos arquivos que serão tocados. Se encontrar problemas sérios (P0/P1), ele avisa e bloqueia até você decidir o que fazer.

Onde fica salvo:

~/beto-vault/projetos/audiencias/plans/ └── task-T04.md ← plano detalhado passo a passo
⚡

Fase 4 — execute

4
Implementar EXATAMENTE o plano
Código → build check → Codex Post-Review → commit → push. Nada fora do plano.
execute <ID>
/projeto audiencias execute T04

O que o Beto faz:

1
Lê o plano da tarefa
Apenas o task-T04.md. Não improvisa nada que não esteja ali
2
Implementa passo a passo
Cria/modifica os arquivos especificados. Se encontrar algo inesperado, documenta e pausa
3
Verifica o build
npm run build após cada arquivo. Não avança com erro de TypeScript
4
Codex Post-Review
Revisão automática dos arquivos modificados. Se achar P0/P1, corrige antes do commit
5
Commit + Push
git add -A → commit com mensagem padrão → push origin main

Onde fica salvo:

~/beto-vault/projetos/audiencias/logs/ └── task-T04-result.md ← o que foi feito, arquivos, como testar
🚫
Execute só implementa o que está no plano
Se o plano não cobre algo que parece necessário, o Beto para e documenta — não improvisa. Se quiser adicionar algo, volte ao plan e atualize.
⚡

Atalhos do dia a dia

next Próxima tarefa — atalho mais usado

Encontra a próxima tarefa disponível (dependências satisfeitas) e faz plan + execute automaticamente.

/projeto audiencias next # Beto vai: # 1. Ler tasks.json # 2. Encontrar T05 (próxima com deps OK) # 3. Fazer plan T05 + execute T05 na mesma sessão # 4. Commitar e reportar progresso
💡
Workflow recomendado
Depois de revisar spec e break, fica só usando /projeto audiencias next em cada sessão. O Beto gerencia a fila de tarefas automaticamente.

Regras importantes

Regra Por quê
Nome do projeto sempre obrigatório
/projeto audiencias spec "..."		 Suporta múltiplos projetos em paralelo sem colisão
/clear entre fases
spec → /clear → break → /clear → plan		 Context window limpo = output muito melhor. Cada fase lê apenas seus arquivos
Revisar spec antes do break
Erros no spec se propagam para todas as fases		 É mais fácil corrigir a spec que refazer 7 tarefas
Execute não improvisa
Se falta algo no plano, para e reporta		 Evita código inesperado em produção
Tarefa falhou 2x → parar
Beto reporta com contexto completo		 Melhor escalar do que entrar em loop

Legenda de status das tarefas

[ ] Pendente
[~] Planejada (plan feito)
[x] Concluída (execute feito)
[v] Validada
[!] Falhou
