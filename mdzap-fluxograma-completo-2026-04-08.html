2. Fluxo Completo do Lead (11 Fases)

FASE 1 — INBOX
Mensagem chega via WhatsApp
Meta Cloud API envia webhook → Edge Function whatsapp-webhook normaliza → INSERT em whatsapp_mensagens + UPDATE whatsapp_conversas → Fire-and-forget POST para n8n mdzap/orquestrador (Julia decide resposta)
Edge Function mdzap/orquestrador whatsapp_mensagens.INSERT whatsapp_conversas.UPSERT Realtime: whatsapp_mensagens
FASE 2 — DETECÇÃO & VISUALIZAÇÃO
Operador vê conversa no MdZap
useMdzapConversas (RPC fn_mdzap_conversas) lista conversas com unread count. Click na conversa → useMdzapMensagens carrega mensagens (50/page, infinite scroll). Se lead não existe → botão "Criar Lead". Se lead existe → LeadPanel abre com dados. Doc Reader pode já ter pré-preenchido pending_lead_data na conversa.
fn_mdzap_conversas useMdzapConversas useMdzapMensagens MdzapConversaList MdzapMsgThread
FASE 3 — CRIAR/VINCULAR LEAD
Criação do lead a partir da conversa
useLeadPanelActions.criarLead() → INSERT em leads com dados do contato → RPC fn_link_conversa_to_lead vincula whatsapp_conversas.lead_id ao novo lead. Coluna CRM inicia como 'novos'. Se Doc Reader pré-preencheu, dados são copiados.
fn_link_conversa_to_lead leads.INSERT useLeadPanelActions coluna_crm='novos'
FASE 4 — PREENCHIMENTO DE DADOS
Operador preenche dados do lead (LeadDadosTab)
Form em LeadDadosTab + LeadDadosFields com campos: nome, cpf, email, telefone, whatsapp, rg, data_nascimento, sexo, profissao, estado_civil, endereço completo. saveLead() faz UPDATE direto em leads. Não passa por webhook.
nome cpf email telefone whatsapp rg data_nascimento sexo profissao estado_civil endereco_logradouro endereco_numero endereco_complemento endereco_bairro endereco_cidade endereco_uf endereco_cep
FASE 5 — AUDITORIA (Bureau)
Disparo de auditoria de restrições no bureau
useLeadPanelActions.triggerCadastrarLead() → RPC rpc_disparar_auditoria → n8n recebe e chama bureau apifull_115 com CPF do lead → resultado cria registros em restricoes automaticamente. Lead move para coluna_crm='em_atendimento'.
rpc_disparar_auditoria n8n: apifull_115 restricoes.INSERT (auto) P1: sem feedback visual de progresso
FASE 6 — GESTÃO DE RESTRIÇÕES
Cadastro e gestão de restrições do lead
LeadRestricoesMdzapTab exibe restrições. RPC fn_restricao_salvar cria/atualiza restrições com: tipo (comprador/devedor/fiador/avalista), empresa credora, valor original, parcelas, datas. Cada restrição pode gerar um processo futuro.
fn_restricao_salvar tipo empresa_credora valor_original contrato data_inclusao data_vencimento orgao_registro
FASE 7 — UPLOAD DE DOCUMENTOS
Sistema de documentos em 3 tiers
LeadDocumentosTab gerencia upload para Supabase Storage. 3 tiers: (1) Documentos centrais do lead (RG, CPF, comprovante), (2) Documentos por restrição (contrato, extrato específico), (3) Documentos por processo (após conversão). RPC fn_doc_append registra metadados.
fn_doc_append fn_doc_get_signed_url fn_doc_delete Tier 1: lead_docs Tier 2: restricao_docs Tier 3: processo_docs P0: Doc Reader webhook NOT IMPLEMENTED
FASE 8 — ASSUMIR / DEVOLVER JULIA
Controle de atendimento humano ↔ IA
Assumir: useMdzapThreadActions.assumirConversa() → webhook mdzap/assumir-conversa → n8n MISSING (webhook não existe!). Frontend faz UPDATE direto em whatsapp_conversas.usuario_id_responsavel.
Devolver: webhook mdzap/devolver-julia → 2 workflows conflitantes no n8n (path collision).
mdzap/assumir-conversa mdzap/devolver-julia P0: assumir-conversa n8n MISSING P0: devolver-julia 2 WFs conflitantes whatsapp_conversas.UPDATE
FASE 9 — MENSAGENS (Envio/Recebimento)
Fluxo completo de envio de mensagens
useMdzapSendMessage: INSERT otimístico em whatsapp_mensagens (status='pending') → webhook mdzap/enviar-mensagem → n8n chama Meta Cloud API → callback atualiza status para 'sent'/'delivered'/'read'. Se falha → status='failed' + toast. Suporta: text, image, audio, video, document, template, link.
mdzap/enviar-mensagem mdzap/enviar-template mdzap/enviar-audio whatsapp_mensagens.INSERT (optimistic) Realtime: status updates P1: enviar-audio é stub
FASE 10 — MOVIMENTAÇÃO CRM
Lead move entre colunas do CRM
Badge dropdown no MdzapThreadHeader permite mover lead entre colunas CRM. UPDATE direto em leads.coluna_crm. Colunas válidas: novos, em_atendimento, aguardando_documentos, aguardando_assinatura, triagem, arquivados, convertido, documentos_pendentes.
MdzapBadgeStatus leads.coluna_crm UPDATE novos em_atendimento aguardando_documentos aguardando_assinatura triagem convertido documentos_pendentes arquivados
FASE 11 — AÇÕES BULK & CONVERSÃO
Arquivar, transferir em massa, converter para cliente
Bulk: mdzap/arquivar-conversas-bulk, mdzap/transferir-conversas-bulk, mdzap/bulk-negociacoes, mdzap/bulk-documentos. Conversão: quando lead atinge coluna_crm='convertido', dados migram para clientes + processos via RPCs dedicadas.
mdzap/arquivar-conversas-bulk mdzap/transferir-conversas-bulk mdzap/bulk-negociacoes mdzap/bulk-documentos fn_lead_converter

5. Agentes IA — Hierarquia & Comportamento

Julia (Orquestradora)

Modelo: Claude Sonnet
Função: SDR principal — qualifica leads, coleta dados, responde WhatsApp.
3 Modos SDR: sdr_leads (qualificação), sdr_docs (coleta documentos), sdr_acordos (negociação).
9 Tipos de Prompt: sdr_principal, sentimento, sugestoes, memoria, fluxo_atendimento, regras, ai_composer, sdr_acordos, sdr_docs
Trigger: mdzap/orquestrador (fire-and-forget da Edge Function)
Tabelas: mdzap_julia_prompts, mdzap_lead_memory, whatsapp_mensagens
Output: INSERT mensagem + UPDATE lead status

Doc Reader

Modelo: Claude Sonnet
Função: Lê documentos enviados pelo lead (fotos de RG, CPF, contracheque). Extrai dados estruturados e preenche pending_lead_data.
Status: NOT IMPLEMENTED — spec E-12 requer, mas nem frontend nem n8n existem.
Trigger: deveria ser mdzap/doc-reader (não existe)
Helper: doc-reader-helpers.ts (102 linhas — só frontend)
Output esperado: preencher LeadDadosFields automaticamente

Memory Agent

Modelo: Claude Sonnet
Função: Extrai e consolida memória da conversa — CPF, nome, empresa credora, valor dívida, situação descrita, documentos enviados.
Status: PARCIAL — webhook mdzap/memoria existe, frontend MdzapLeadMemoryPanel funciona.
Trigger: mdzap/memoria (MdzapLeadMemoryPanel:66)
Tabela: mdzap_lead_memory
Output: dados estruturados extraídos da conversa

ICP / Telemetria

Modelo: Claude Sonnet
Função: Calcula Ideal Customer Profile score. Analisa: valor dívida, tipo restrição, perfil socioeconômico, probabilidade de sucesso jurídico.
Status: ATIVO
Trigger: mdzap/icp-analise (useMdzapIcp:66)
Tabela: mdzap_icp_analises
Dashboard: MdzapTelemetriaPanel + MdzapIcpTab
