Bug Confirmado #2 — Context CRM Oculta Restrições Vinculadas

Bug ativo no context CRM:

Quando o ClienteDrawer abre em context=crm, showProcessoStatus = false, então apenasSemProcesso = !false = TRUE. Isso filtra apenas restrições sem processo vinculado. Como 98,3% das restrições têm processo_id, praticamente nenhuma aparece no contexto CRM.

// context=crm → isCrm=true
showProcessoStatus = !isCrm && !isDistribuicao = false

// Na tab:
apenasSemProcesso = !showProcessoStatus = true

// Resultado: fn_restricoes_listar com p_apenas_sem_processo=TRUE
// Das 4.354 restrições com processo_id → NENHUMA retornada
// Apenas as 73 sem processo_id seriam visíveis no CRM
Context showProcessoStatus apenasSemProcesso Restrições visíveis Correto?

Nota: para o contexto CRM, mostrar apenas sem processo pode ser o comportamento INTENCIONAL (o CRM quer ver restrições para distribuir). Para context=distribuicao é definitivamente correto. A questão é se no CRM também se quer ver restrições já distribuídas — isso precisa de validação com o CTO.

Causa Raiz Principal — Déficit de Dados (71,6%)

Problema de Dados, não de Código:

Para 3.112 clientes (71,6%), a aba Restrições mostra corretamente "Nenhuma restrição cadastrada" porque genuinamente não existe nenhuma linha na tabela restricoes vinculada a esses clientes.

Por que 71,6% dos clientes não têm restrições?

  • Todos os clientes foram criados em março de 2026 (migração ADVBOX)
  • A migração de restrições cobriu apenas 1.235 clientes dos 4.347
  • Os processos desses 3.112 clientes têm campos legados (restricao_orgao_nome, restricao_valor, etc.) mas sem linha correspondente na tabela restricoes
  • 2.349 processos têm restricao_orgao_nome preenchido mas restricao_id = null
  • O ADVBOX não exporta os dados de "anotações de restrição" no mesmo endpoint dos processos

Proposta de Fix

Fix 1 (dados) — Migração retroativa das restrições

Criar script para extrair dados de restrição dos campos legados (restricao_orgao_nome, restricao_valor, restricao_tipo, restricao_contrato) dos processos e inserir linhas na tabela restricoes vinculadas a cliente_id e processo_id.

Arquivo: nova migration SQL

-- Migração: criar restrições a partir dos dados legados nos processos
-- Escopo: 2.349 processos com restricao_orgao_nome não nulo e restricao_id nulo
-- REQUER APROVAÇÃO CTO antes de executar em produção

INSERT INTO restricoes (
  organizacao_id, cliente_id, processo_id,
  orgao_nome, empresa_nome,
  tipo, codigo_contrato, data_restricao,
  valor, processavel, created_by
)
SELECT
  p.organizacao_id,
  p.cliente_id,
  p.id as processo_id,
  p.restricao_orgao_nome,
  p.empresa_nome,
  p.restricao_tipo,
  p.restricao_contrato,
  COALESCE(p.restricao_data_new, p.restricao_data),
  COALESCE(p.restricao_valor_new, p.restricao_valor),
  true as processavel,
  p.created_by
FROM processos p
WHERE
  p.restricao_orgao_nome IS NOT NULL
  AND p.restricao_id IS NULL
  AND p.cliente_id IS NOT NULL
  AND p.deleted_at IS NULL
  AND p.organizacao_id = '55a0c7ba-1a23-4ae1-b69b-a13811324735';

-- Após inserção: atualizar processos.restricao_id
UPDATE processos p
SET restricao_id = r.id
FROM restricoes r
WHERE
  r.processo_id = p.id
  AND p.restricao_id IS NULL
  AND r.created_at > NOW() - INTERVAL '1 minute';
-- Limitar a restrições recém-criadas para segurança
Fix 2 (código) — Verificar intenção do filtro CRM

O filtro apenasSemProcesso=true no context CRM é intencional para mostrar apenas restrições pendentes de distribuição. Porém, depois que uma restrição é distribuída (recebe processo_id), ela desaparece da aba CRM. Se a intenção é mostrar TODAS as restrições também no CRM, alterar o ClienteDrawerTabRestricoes para sempre usar apenasSemProcesso=false e usar o badge de "Processo vinculado" para indicar o status.

Resumo Executivo para o CTO

O que está acontecendo

Quando um usuário abre o ClienteDrawer em /clientes e clica na aba "Restrições", para 71,6% dos clientes (3.112 de 4.347) a aba mostra "Nenhuma restrição cadastrada". O código está correto — a query RPC funciona, os parâmetros são passados corretamente. O problema é que esses clientes simplesmente não têm linhas na tabela restricoes.

Por que não têm dados

A migração ADVBOX de março/2026 importou processos com os dados de restrição embutidos nos campos legados do processo (restricao_orgao_nome, restricao_valor, etc.) mas não criou linhas na tabela restricoes. Resultado: 2.349 processos têm os dados mas estão "invisíveis" para a aba de restrições.

Ação necessária

  • P0 Aprovar e executar migration SQL para criar linhas em restricoes a partir dos campos legados dos processos (~2.349 linhas)
  • P1 Revisar se o filtro apenasSemProcesso=true no context CRM é intencional ou não
  • P2 Verificar pipeline ADVBOX para garantir que novos processos criam restrições na tabela desde o início
