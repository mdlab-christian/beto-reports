Lead → Cliente + Processos
O pipeline CRM transforma um contato bruto (lead) em um cliente com processos rascunho prontos para distribuicao. Cada etapa e atomica e pode falhar independentemente sem comprometer as anteriores. O mesmo pipeline sera replicado no MDZap, reutilizando RPCs e hooks compartilhados.
Camadas de responsabilidade
Frontend (React)
Exibe dados, coleta input, dispara acoes via hooks. Nunca contem logica de negocio.
Backend (Supabase)
Guarda dados, valida via RPC SECURITY DEFINER, garante atomicidade e RLS.
Automacoes (n8n)
Executa integracoes externas (bureaus), orquestra fluxos multi-step, consultas IA.

Funcao: findBestMatch(credor, empresas)
Prioridade Algoritmo Threshold Confianca
1 Match exato normalizado em nome_variacoes score = 1.0 Alto
2 Fuzzy tokenSimilarity em nome/razao_social/nome_fantasia score ≥ 0.75 Alto
3 Fuzzy em nome_variacoes (ja confirmado antes) score ≥ 0.6 Alto
4 Score entre thresholds 0.45 ≤ score < threshold Medio
5 Nenhum match encontrado score < 0.45 Nenhum
// normalizeText: lowercase, remove acentos, remove nao-alfanumericos function normalizeText(text: string): string[] { return text .toLowerCase() .normalize('NFD') .replace(/[\u0300-\u036f]/g, '') .replace(/[^a-z0-9\s]/g, '') .split(/\s+/); } // tokenSimilarity: Jaccard-like (intersecao / max dos sets) function tokenSimilarity(a: string[], b: string[]): number { const setA = new Set(a); const setB = new Set(b); const intersection = [...setA].filter(x => setB.has(x)); return intersection.length / Math.max(setA.size, setB.size); }

Regra critica: Restricoes da Boa Vista recebem tratamento especial. Alem dos itens individuais (1 por credor), o sistema adiciona 1 item EXTRA agrupado com a soma de todas restricoes Boa Vista.

const boaVistaKeywords = ['scpc/boa vista', 'scpc', 'boa vista', 'boavista']; // Qualquer match case-insensitive -> true

RegraDescricao
1. IndividuaisTODOS os itens ficam individuais (1 por credor) — NAO agrupa
2. Item Extra BV Se tem restricoes Boa Vista → adiciona 1 item EXTRA agrupado:
empresa = "Boa Vista" | valor = SOMA total BV | processavel = SEMPRE true | tipo = 'comprador'
Exemplo16 restricoes apifull_115 → 16 individuais + 1 Boa Vista extra = 17 total

4.5 Salvamento (Loop) RPC

Para cada item com incluir=true
01
Criar empresa se necessario

Se empresa_id e null E tem empresa_nome → INSERT nova empresa. processavel herda do item. status_aprovacao baseado em processavel.

02
RPC fn_restricao_salvar

SECURITY DEFINER. Recebe todos os campos da restricao e persiste atomicamente.

03
Auto-aprendizado nome_variacoes

Se o usuario selecionou empresa existente MAS credor tem nome diferente → adiciona credor em empresa.nome_variacoes. Na proxima vez, o matching sera automatico.

04
Vincular auditoria ao lead

Apos todos salvos → UPDATE auditorias SET lead_id = leadId. Garante rastreabilidade.

4.6 Por que nao quebra: RPC fn_restricao_salvar usa SECURITY DEFINER (bypassa RLS). Auto-aprendizado nome_variacoes faz o sistema ficar mais inteligente a cada uso. Criacao automatica de empresa nova quando credor desconhecido. Item por item no salvamento (falha parcial nao perde tudo). Vinculacao auditoria → lead garante rastreabilidade.

Fluxo Atomico (1 transacao PostgreSQL)
01
Validar lead — Existe, nao convertido, esta em 'triagem'. Erros: ALREADY_CONVERTED, INVALID_STAGE
02
Registrar opt-outs — INSERT em clientes_empresas_optout para restricoes rejeitadas
03
Criar ou reutilizar cliente — Busca por CPF. Se encontra → atualiza advogado. Se nao → INSERT novo cliente
04
Criar processos — Para cada restricao selecionada → INSERT processo (status='rascunho', origem_dados='crm_triagem')
05
Marcar lead — convertido=true, arquivado=true, coluna_crm='arquivados'
06
Log interacao — INSERT leads_interacoes (historico de operacoes)
// Retorno da RPC: { "success": true, "data": { "cliente_id": "uuid", "cliente_existente": false, "processos_ids": ["uuid1", "uuid2", "uuid3"] } }
5.3 Por que nao quebra: Transacao atomica (tudo ou nada). ALREADY_CONVERTED previne race condition. INVALID_STAGE garante estagio correto. Reutilizacao de cliente existente por CPF. SECURITY DEFINER bypassa RLS para INSERT multi-tabela.

Secao 8 — Por Que Nao Quebra: Resumo Consolidado

  1. Deduplicacao de leads (CPF/telefone) previne duplicatas no Kanban
  2. Validacao CPF 11 digitos em 3 camadas (frontend, RPC, n8n) — nunca chega invalido no bureau
  3. Transacoes atomicas PostgreSQL nas RPCs SECURITY DEFINER — tudo ou nada
  4. Dupla estrategia Realtime + Polling para status de auditoria — se Realtime cair, polling pega
  5. Progress bar visual para UX durante espera — usuario nao fica no escuro
  6. Auto-aprendizado de nome_variacoes — matching melhora progressivamente com cada uso
  7. Criacao automatica de empresa nova quando credor e desconhecido — nunca trava
  8. fn_restricao_salvar com SECURITY DEFINER — bypassa RLS, evita problemas de permissao
  9. Race condition protegida com ALREADY_CONVERTED — 2 atendentes nao convertem o mesmo lead
  10. Validacao de estagio INVALID_STAGE — lead deve estar no estagio correto para avancar
  11. Reutilizacao de cliente existente por CPF — evita duplicar clientes no sistema
  12. Salvamento item por item nas restricoes — falha parcial nao perde itens ja salvos
  13. Soft delete em todas as tabelas — nunca perde dados, sempre recuperavel
  14. AbortController timeout 60s nos webhooks — nao trava indefinidamente
  15. Envelope canonico padronizado com request_id — rastreabilidade completa de ponta a ponta
