CRITICAL ISSUES (P0) — FIX IMMEDIATELY
=================================================================

P0.1: MISSING deleted_at COLUMNS IN ALL 21 TABLES
────────────────────────────────────────────────
Status: CRITICAL BLOCKER
Tables affected: users, advogados, organizacao_setores, + 18 others
Severity: Data integrity breach
Impact: Hard deletes cause FK constraint violations

Evidence:
  - TypeScript types (src/types/supabase.ts) show NO deleted_at field
  - Frontend code doesn't filter .is('deleted_at', null)
  - MdFlow pattern requires soft deletes on all business tables

Risk scenario:
  1. Admin deletes user via frontend
  2. DELETE hits FK constraint (processos.responsavel_id references user)
  3. User cannot be deleted; system accumulates orphaned records
  4. No audit trail of deletion intent

Fix timeline: 2 hours
Fix complexity: LOW (schema migration + data migration)

P0.2: MISSING updated_at TRIGGERS ON 12+ TABLES
────────────────────────────────────────────────
Status: CRITICAL BLOCKER
Tables affected: atividades_tipos, config_etiquetas_catalogo, 
                 contas_bancarias, cartoes_credito, 
                 controller_dominios, crm_*, docs_*, 
                 mdzap_ai_modules, whatsapp_numeros
Severity: Audit trail failure
Impact: Cannot track when configuration changes were made

Evidence:
  - Supabase type definitions include updated_at column
  - No trigger set_updated_at defined for these tables
  - Mutations via UPDATE don't auto-set updated_at

Risk scenario:
  1. Admin creates new etiqueta at 14:32
  2. Updated_at column remains NULL or old value
  3. Audit query: "When was this changed?" → Unknown
  4. Compliance gap: No timestamped change log

Fix timeline: 1 hour
Fix complexity: LOW (trigger creation for each table)

P0.3: RLS POLICY VERIFICATION NEEDED ON users TABLE
───────────────────────────────────────────────────
Status: CRITICAL BLOCKER
Table: users
Severity: Multi-tenant data leak risk
Impact: Cross-org user access possible without DB-level enforcement

Evidence:
  - Frontend has client-side admin check (navigates away if not admin)
  - Client-side guard is bypassable (JWT modification, direct API call)
  - RLS policy required: organizacao_id = public.get_org_id()

Risk scenario:
  1. Attacker modifies JWT to change organizacao_id
  2. Frontend guards don't prevent API call
  3. Without RLS, Supabase returns user records from other orgs
  4. Data leak: emails, phone numbers, PIX keys exposed

Fix timeline: 30 minutes
Fix complexity: LOW (verify existing policy exists)
