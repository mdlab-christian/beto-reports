E5
Processamento: Comprovante de Residencia Subagente Docs Julia
  • Verifica legibilidade
  • Verifica se nome OU CPF do lead esta no documento
NAO esta no nome: Julia sugere boleto banco digital (NAO extrato) — mostra endereco no nome. Sem boleto: instrui declaracao de terceiro.
Modelo declaracao terceiro:
"Eu, [Nome Terceiro], portador do CPF [X], declaro que [Nome Lead], portador do CPF [Y], reside comigo nesta moradia."
OBRIGATORIO: documento de identificacao do terceiro junto
Está no nome: extrai endereco completo → popula no sistema
E6
Pre-Auditoria Julia
  • Confirma recebimento dos documentos
  • Mensagem: "Perfeito! Vou encaminhar para nossa equipe realizar a auditoria do seu CPF. Em breve retorno com as informacoes! 🔍"
  • Sistema: awaiting_human=true → badge "Auditoria Pendente" aparece na lista de conversas
E7
Auditoria Humano — Atendente
  • Atendente ve badge "🔍 Auditoria Pendente" no MdZap
  • Faz auditoria via Boa Vista (API Full)
  • Cadastra lead no CRM com coluna: em_atendimento
  • Popula dados pessoais (ja disponiveis via docs) + restricoes
  • Campo profissao pode faltar → preenchido depois via CTPS
  • Atendente envia resultado ao lead (modo humano)
E8
Resultado da Auditoria Atendente envia Julia retoma
NOME LIMPO: PDF do sistema (comprovante de auditoria) → parabenizar lead → nao mencionar Indique Ganho → encerrar cordialmente
NOME SUJO: Mencionar empresas por nome (NUNCA valores) → quando empresa se repete: citar frequencia ("39 restricoes da RGE Sul") → demonstrar especialidade → prosseguir para CTPS

Contrato ABERTO: cargo → profissao do lead
Contrato FECHADO: Julia pergunta situacao:
· Empregado em outro lugar → pedir novo documento
· Autonomo → nao precisa comprovante (Midas faz renda receita via analise IR)
· Desempregado → Julia cadastra profissao="desempregado"
E10
Renda Receita + Procuracao Humano Julia
  • Renda receita: Midas faz analise IR externamente (processo manual)
  • Se lead nao declarou IR: Julia solicita declaracao IR
  • Humanos geram procuracao no MdFlow (ZapSign)
  • Julia move lead para aguardando_assinatura
  • Julia faz follow-ups: apos 1h sem resposta → segue crm_followup_config
  • Apos assinatura: "Seu caso foi encaminhado ao nosso escritorio de advocacia parceiro. Obrigado pela confianca na Midas! 🎉"

Decisoes Finais P1–P10

Todas respondidas na reuniao 08/04
P1 Criacao do lead no CRM ⚠️ INVESTIGAR
Pre-cadastro: dados ficam na conversa (whatsapp_contatos + metadata) ate engajamento real.
Lead so e criado no CRM quando: lead enviar documentos OU confirmar interesse de forma definitiva.
Risco evitado: leads de spam do Facebook Ads poluindo o banco.
Schema atual: whatsapp_contatos + whatsapp_conversas (lead_id nullable).
Investigar: como mostrar dados da conversa no painel sem lead_id criado.
P2 Coluna CRM inicial ✅ DEFINIDO
em_atendimento diretamente — leads WhatsApp ja estao em atendimento com Julia.
Observacao: verificar impacto nos dashboards de "novos leads".
P3 Subagente de documentos ✅ DEFINIDO
n8n workflow separado — webhook: document-received (criado, nunca implementado).
Fluxo: media_id → download Meta → Claude Vision → JSON → atualiza lead → notifica Julia
P4 Campo profissao ✅ DEFINIDO
ADD COLUMN profissao TEXT em leads.
cargo = profissao (mesmo campo, sem duplicar).
Nao criar input separado para cargo — profissao basta.
P5 Notificacao auditoria ✅ DEFINIDO
awaiting_human=true + badge "🔍 Auditoria Pendente" na lista de conversas MdZap.
Quando Julia envia mensagem pre-auditoria → n8n seta awaiting_human=true
P6 Julia acessa restricoes ✅ DEFINIDO
n8n consulta leads.restricoes e injeta no system prompt do Claude.
Query: SELECT restricoes FROM leads WHERE id = {lead_id}
P7 Quem envia resultado auditoria ✅ DEFINIDO
ATENDENTE SEMPRE ENVIA (modo humano).
Julia assumir esse passo = feature futura — nao implementar agora.
P8 PDF nome limpo ⚠️ INVESTIGAR
PDF gerado pelo sistema via botao de auditoria (comprovante de auditoria).
Boa Vista API Full e a unica integrada — Serasa e SPF ainda nao.
Investigar: fluxo completo do botao de auditoria + como funciona com CRM.
P9 Renda Receita ✅ DEFINIDO
Processo manual externo por enquanto.
Julia: apenas solicita declaracao IR se faltar — nao executa nada automaticamente.
P10 Indique Ganho ✅ DEFINIDO
Julia NAO menciona nada — feature nao pronta.
Implementar quando Indique Ganho estiver pronto no sistema.

Diagnostico Midia Inbound

4 causas identificadas
Causas por probabilidade
C1
n8n nao executa fluxo media_id → URL temporaria → download → Storage
Causa mais provavel — workflow nunca implementado
60%
C2
Upload para Storage nao acontece / bucket errado
Inconsistencia: "mdzap-media" vs "chat-media"
25%
C3
Bucket privado + getPublicUrl() = 403
URL gerada invalida para acesso externo
10%
C4
URL temporaria da Meta salva diretamente
Expira em ~5min — invalida para storage permanente
5%
Fluxo correto — implementar exatamente assim
Meta webhook → media_id → GET graph.facebook.com/v20.0/{media_id} Authorization: Bearer {META_TOKEN} → response.url (URL temporaria ~5min) → GET {response.url} Authorization: Bearer {META_TOKEN} → binario do arquivo → POST Supabase Storage bucket/{org_id}/{conversa_id}/{timestamp}-{filename} → storage_url permanente → UPDATE whatsapp_mensagens SET media_url = storage_url WHERE id = {mensagem_id}
⚠️ ALERTA: inconsistencia no nome do bucket — diagnostico diz "mdzap-media", spec anterior diz "chat-media". Confirmar com Christian ANTES de implementar.
