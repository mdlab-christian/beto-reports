Issues P0 — Bloqueadores

P0 Inconsistência: arquivada vs status_analise
Problema: Tabela referencia campo arquivada em migration 20260305100001 (WHERE arquivada = false), mas também usa status_analise com valores 'pendente' | 'arquivada' | NULL
Impacto: Lógica de negócio quebrada. Trigger filtra status_analise <> 'arquivada' mas índice filtra arquivada = false
Raiz: Qual é verdadeiramente o flag de arquivo? Há duplicação semântica.
P0 RLS Policy não validado
Problema: Não foi possível validar RLS policies. Supabase Management API indisponível.
Crítico: MdFlow obriga POLICY "decisoes_org_isolation" ON decisoes USING (organizacao_id = public.get_org_id())
Risco: Possível data leak multi-tenant se policy estiver faltando
P0 Índices em organizacao_id não validados
Esperado:
CREATE INDEX idx_decisoes_org ON decisoes(organizacao_id); CREATE INDEX idx_decisoes_org_active ON decisoes(organizacao_id) WHERE deleted_at IS NULL;
Status: Não foi possível validar via MCP. Aplicar imediatamente.

Issues P1 — Importantes

P1 FK intimacao_id com semântica duvidosa
Problema: Migration 20260306000001 adiciona intimacao_id FK, mas RPC 20260305100003 também faz UPDATE intimacoes SET decisao_id (relação bidirecional)
Questão: Uma intimação pode ter múltiplas decisões? Ou uma decisão referencia múltiplas intimações?
Padrão MdFlow: FKs devem ter semântica clara (1:N vs N:N com tabela de join)
P1 status_analise sem CHECK CONSTRAINT claro
Problema: Migration referencia valores 'pendente' | 'arquivada', mas não há CHECK CONSTRAINT documentado
Risco: Inserção de valores inválidos via INSERT direto (sem passar pela RPC)

Issues P2 — Melhorias

P2 Campos de cálculo permitindo NULL
Problema: valor_total_calculado, data_calculo, calculo_memoria podem ser NULL indefinidamente
Melhoria: Adicionar DEFAULT ou computed column se são sempre calculados via rpc_calcular_valor_atualizado
Severidade: Apenas DX. Não bloqueia funcionamento.

SQL de Correção — Pronto para Aplicar

P0.1 — RLS Policy (OBRIGATÓRIO)

ALTER TABLE decisoes ENABLE ROW LEVEL SECURITY; DROP POLICY IF EXISTS "decisoes_org_isolation" ON decisoes; CREATE POLICY "decisoes_org_isolation" ON decisoes FOR ALL USING (organizacao_id = public.get_org_id()) WITH CHECK (organizacao_id = public.get_org_id());

P0.2 — Índices Obrigatórios

CREATE INDEX IF NOT EXISTS idx_decisoes_org ON decisoes(organizacao_id); CREATE INDEX IF NOT EXISTS idx_decisoes_org_active ON decisoes(organizacao_id) WHERE deleted_at IS NULL; CREATE INDEX IF NOT EXISTS idx_decisoes_valor_total ON decisoes(organizacao_id, valor_total_calculado) WHERE valor_total_calculado IS NOT NULL AND (status_analise IS NULL OR status_analise <> 'arquivada');

P0.3 — CHECK Constraint em status_analise

ALTER TABLE decisoes ADD CONSTRAINT decisoes_status_check CHECK (status_analise IS NULL OR status_analise IN ('pendente', 'arquivada', 'analisada')) NOT VALID; VALIDATE CONSTRAINT decisoes_status_check;

P0.4 — Trigger updated_at (se faltando)

CREATE TRIGGER IF NOT EXISTS trg_decisoes_updated_at BEFORE UPDATE ON decisoes FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

P0.5 — Esclarecer semântica de arquivada vs status_analise (DECISION)

-- Opção A: RECOMENDADA — Remover "arquivada" e usar APENAS status_analise ALTER TABLE decisoes DROP COLUMN IF EXISTS arquivada CASCADE; -- Opção B: Remover "status_analise" e usar APENAS "arquivada" -- ALTER TABLE decisoes DROP COLUMN IF EXISTS status_analise CASCADE; -- ALTER TABLE decisoes ADD COLUMN arquivada BOOLEAN DEFAULT false; -- ALTER TABLE decisoes ADD CONSTRAINT decisoes_arquivada_check CHECK (arquivada IN (true, false));
